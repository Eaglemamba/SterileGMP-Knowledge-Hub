Appendix 6 — Infrastructure Security (p84–p90)

原文內容 Original Text

1. Introduction

A defined level of infrastructure security is required to meet business purposes and to satisfy external regulations. Lack of security may compromise availability of applications and services, record integrity and confidentiality, reputation with stakeholders, and may lead to unauthorized use of systems that would impact on product quality.

A company should take cultural and practical aspects into consideration when defining the rigor of its approach. Companies may adopt different approaches for specific infrastructure elements, based, e.g., on criticality and risk.

2. Infrastructure Security Management

Infrastructure security management includes all the policies, procedures, requirements, training and audit programs, etc. that a company may define appropriate to safeguard the infrastructure, including information assets.

As a minimum, regulated companies need to satisfy the applicable GxP requirements. Relevant documents include the PIC/S Guide, which builds on ISO/IEC 17799:2000 Information Technology – Code of Practice for Information Security Management. The 21 CFR Part 11 contains security requirements focused on electronic records and signatures. RFC 2196, "Site Security Handbook," is another useful source.

Security controls, in general, are considerably cheaper and more effective if incorporated at the requirements specification and design stages.

2.1 ISO/IEC 17799 — Code of Practice for Information Security Management

Information security is characterized in the standard as the preservation of:

  • Confidentiality: ensuring that information is accessible only to those authorized to have access
  • Integrity: safeguarding the accuracy and completeness of information and processing methods
  • Availability: ensuring that authorized users have access to information and associated assets when required

The ISO/IEC 17799:2000 is a risk-based method for assessing, evaluating, and managing risks. It provides a framework for developing a security program with 10 sections:

  1. Security Policy
  2. Organizational Security
  3. Asset Classification and Control
  4. Personnel Security
  5. Physical and Environmental Security
  6. Communications and Operations Management
  7. Access Control
  8. System Development and Maintenance
  9. Business Continuity Management
  10. Compliance

Not all controls will be relevant to all infrastructures. The use of ISO/IEC 17799 as a guide should be adapted to the circumstances of each organization.

2.2 ISO/IEC 17799 and Regulations

Regulatory guidance recommends consideration of these standards, but does not require organizations to be formally accredited. Companies should define a list of ISO/IEC 17799 issues relevant to their infrastructure and map existing policies against applicable sections, using internal audits to demonstrate control to regulatory authorities.

導師解析 Tutorial Commentary

核心概念：CIA 三角 The CIA Triad

Confidentiality（機密性）：資料只有授權人員可存取。GxP 系統中，批次紀錄不得被未授權人員讀取或下載。

Integrity（完整性）：資料必須準確且完整，任何修改都要留下稽核軌跡。這直接對應 21 CFR Part 11 的電子記錄要求。

Availability（可用性）：授權使用者在需要時必須能存取系統。若生產線因 IT 系統停機無法取得批次記錄，即構成可用性失效。

比喻說明 Analogy

把 CIA 三角想像成醫院的病歷管理系統。機密性 = 只有主治醫生能看完整病歷；完整性 = 任何人修改病歷都留記錄；可用性 = 急診時隨時能調出病歷。三者缺一不可，就像無菌製造中的環境控制——少了任何一環，產品安全就有風險。

重點提示 Key Notes

ISO/IEC 17799 是推薦框架，法規不要求正式認證，但稽查時需要能展示「公司已將此框架映射至自身政策並完成內稽」。這是合規證明的基礎，不是選擇題。

「安全控制在需求規格設計階段納入，成本最低、效果最好」——這條原則與 GAMP 的 V-model 一致：後期補救的代價遠高於前期設計。

2.3 Technical versus Procedural Issues

Management should establish an administration obligated to define and monitor an infrastructure security program. Controls include:

  • Administrative controls: policies, standards, procedures, guidelines, security awareness training, incident management
  • Technical controls: access control mechanisms, password and resource management, identification and authentication methods, security devices
  • Physical controls: controlling individual access into facilities, locking systems, removing unnecessary floppy disk or CD-ROM drives, protecting facility perimeter, monitoring for intrusion

2.4 Security Incident Management

An incident management procedure should be established as effective communication channels are essential in the quick identification and mitigation of a specific threat. Seemingly isolated incidents may not receive an appropriate level of attention unless they are centrally reported. For instance, an account being locked due to a forgotten password may become more suspicious if the same event occurs repeatedly.

2.5 Intrusion Detection

Intrusion Detection Systems (IDSs) in the form of hardware or software enhance the security of the perimeter of internal networks. Determined intruders may penetrate firewalls, and attacks do not come only from external sources. By monitoring suspicious activity, alarms can be sounded earlier. A well-maintained pattern database and regular review of IDS summary reports ensure efficacy.

2.6 Vulnerability Management

Vulnerability scanners or adequate manual procedures should be applied to determine weaknesses and gaps in security configurations. Corrective actions include: application of patches, changes to security settings, harnessing the network topology.

A practical approach is to start with a complete inventory of systems, identify a small number of reliable sources of security alerts, assess and prioritize each threat. With numerous new threats identified each day, it is often more appropriate to patch during scheduled maintenance (weekly or monthly) rather than daily.

2.7 Anti-Virus Shield Update

Most anti-virus applications can initiate totally automatic updates, but some companies prefer controlled daily updates. Configuration Management practices are required, recording which updates were applied to which systems, and at what date and time.

Two options for remote/off-site clients: (1) controlled distribution from internal IT — allows testing but inhibits rapid response; (2) automatic updates from anti-virus supplier — poses challenges for Configuration Management. A Risk Assessment comparing untested updates vs. leaving clients unprotected may be warranted.

2.8 Public Key Infrastructure

Where a company needs features such as digital signatures, it may implement a Public Key Infrastructure (PKI), or use PKI services from external organizations. Digital signatures secure authenticity, non-repudiation, integrity, and confidentiality of external messaging. Users need certificates from a trusted third party or the company's own Certification Authority (CA).

2.9 Origin of Software

The origin of all software related to qualified infrastructure platforms should be known. If open source software is used, ensure a reputable supplier supports the product and accepts responsibility for maintaining it. Software from non-reputable or unknown sources should be avoided.

重點提示 Key Notes

帳號重複鎖定是資安事件，不是 IT 小問題。單次鎖定可能是忘記密碼，但重複發生可能是暴力破解攻擊。集中通報機制讓孤立事件變成可見的模式——這與 GMP 的品質趨勢分析原理相同。

防毒軟體更新的兩難：受控更新可測試但反應慢；自動更新反應快但 Configuration Management 難追蹤。GxP 系統優先選受控更新，需以風險評估文件化決策理由。

開源軟體不等於免費風險：若軟體來源不明或無供應商維護承諾，在 GxP 環境中等同於使用未驗證的原材料——合規風險極高。

3. Upgrades and Patches — Balancing Qualification and Security

Changes to IT Infrastructure platforms require formal change control processes, documentation, and testing. These processes take time, especially where individual application tests are required, and particularly for GxP applications.

Where changes are required to address security vulnerabilities, applying the full change control and testing processes prospectively may present an unacceptable risk to the business in terms of security. However, direct application of any upgrades/patches without change control and testing may compromise the qualified status of the IT Infrastructure and present compliance issues.

First, the scope of the change should be understood (Network Operations, Clients, Servers, Firmware/Hardware, Data Management Software, Operating System, Infrastructure Utility Systems). Second, the urgency of the change should be established.

3.1 Urgency of Security Update

Generally, the urgency of application of a change falls into one of three criticality categories:

  1. Emergency Fix: should be applied as soon as possible, where the security vulnerability presents immediate and real threats to the business
  2. Urgent Update: probable threat, fix should be applied within a specified time scale
  3. Update: no immediate threat, consider including within next scheduled platform upgrade

3.2 Compatibility of Security Updates

Security patches are usually available only for a limited number of product versions; suppliers of interacting software may not provide compatible versions, creating a dilemma for system owners. Where critical security patches cannot be installed, system managers may have to resort to procedural controls or even isolate the system from other systems or from the corporate network.

Further guidance on managing security upgrades and patches is given in Appendix 7.

重點提示：緊急補丁仍需變更控制

Emergency Fix 可以「先做後補文件」，但不是「做了就算」。事後補做的變更控制文件必須說明：為何緊急、評估了什麼風險、誰授權、做了什麼事後監控。這是 GxP 系統「qualified status」維護的底線。

CDMO 現場常見錯誤：在 21:00 緊急打了補丁，隔天沒有任何文件記錄，直到稽查才被發現——這是偏差，不是緊急程序。

比喻說明 Analogy

緊急補丁就像醫院的緊急手術——不能等標準排程，但手術後仍要寫完整手術記錄、填偏差報告、追蹤恢復狀況。「緊急」改變的是時序，不是文件化的義務。

補丁相容性困境 Compatibility Dilemma

當關鍵安全補丁無法安裝（因系統版本太舊或應用程式不相容），選項包括：

  • 程序性控制（Procedural Controls）：限制網路存取、加強監控、限制使用者操作
  • 網路隔離（Isolation）：將系統從企業網路斷開，降低攻擊面

這兩個選項都必須透過風險評估文件化，並在 CAPA 中追蹤長期解決方案。

Appendix 7 — Upgrade and Patch Management (p91–p94)

原文內容 Original Text

1. Fundamental Principles

Regardless of the criticality of an upgrade, the following aspects should be considered:

  • Change Control: Change control documentation should be raised to cover application of the upgrade/fix. For emergency changes, this may be retrospective (but within a minimum period of time following application).
  • Configuration Management: Records should be maintained; documentation recording versions and patch levels of platforms should be accurately maintained. For emergency changes, documentation may be retrospectively updated.
  • Communication: All owners of critical applications should be made aware of the requirement for an upgrade prior to its application.
  • Incident Monitoring: An incident monitoring process should be in place. Where emergency changes are applied with minimal or no specific application testing, particular attention should be given to incident monitoring and reporting.

2. Upgrade Strategy

The upgrade strategy should be based on a technical assessment and an impact assessment:

  • The IT group should perform a technical assessment: whether the upgrade is minor, medium, or major based on complexity and possible impact on applications.
  • All owners of critical applications should perform an impact assessment on their systems.

Issues can arise when implementing patches to platforms where the base application and/or operating system have not been upgraded for an extended period and lag behind the current version. Installation of an emergency upgrade may be compromised — resulting in either the application being unable to run or the patch being unable to be applied.

Systems that have been removed from operational use but retained as read-only systems for record/data retrieval (partially decommissioned) may present issues, particularly where product support is no longer available. Consideration should be given to isolating or segregating such systems.

3. Level of Application Testing

Four general levels of testing that may be applied to applications following a platform upgrade:

  1. No Testing
  2. Confirmation Testing: A minimal level of documented testing to verify that an application runs on an updated platform.
  3. Confidence Testing: Documented testing against pre-defined specifications that consist of a representative subset of functional tests on the updated platform.
  4. Full Testing: Full regression testing of the application on the updated platform.

For emergency changes (e.g., critical security update), it is likely that no prospective testing will be applied. For business critical and GxP applications, enhanced incident monitoring should be applied, and a documented justification made until retrospective testing or final impact evaluation is completed.

Where the update is not an emergency: minor updates → Confirmation Testing only may be appropriate; significant updates or highly critical GxP systems → Confidence Testing as minimum; major updates with likely application impact → Full regression testing.

The two-stage Risk Assessment process from GAMP 4 (assessing impact of failure, likelihood of failure, probability of detection) is a good tool for determining appropriate testing level.

4. Global/Multi-Site Systems

For systems used at multiple locations with worldwide standardization of platforms, an efficient approach is to perform testing at a single location and use this as the basis of reduced or no testing at other locations.

導師解析 Tutorial Commentary

核心概念：四級應用程式測試層級

Level 1 — No Testing（無測試）：僅限緊急情況，必須搭配強化事件監控和事後測試計畫。絕非「永久免測」。

Level 2 — Confirmation Testing（確認測試）：最低限度文件化測試，驗證應用程式在更新後平台仍能執行。適用：技術評估為「輕微」的升級。

Level 3 — Confidence Testing（信心測試）：以預定規格執行代表性功能子集測試。適用：顯著升級，或高度關鍵 GxP 系統。

Level 4 — Full Testing（完整回歸測試）：全面回歸測試。適用：主要升級且對應用程式有可能造成影響。

測試層級選擇決策矩陣

升級性質     | GxP/關鍵系統？ | 建議測試層級
-------------|----------------|------------------
緊急補丁     | 是/否          | Level 1 + 強化監控
輕微升級     | 否             | Level 2
輕微升級     | 是             | Level 2~3
顯著升級     | 是/否          | Level 3 (最低)
主要升級     | 是             | Level 4

比喻說明 Analogy

四級測試就像汽車出廠前的品質查驗等級：緊急出廠（Level 1）= 只看引擎能不能發動就交車；確認測試（Level 2）= 還會試踩煞車；信心測試（Level 3）= 在測試跑道跑幾圈；完整測試（Level 4）= 全面安全認證測試。交的車越貴、乘客越重要，測試等級越不能省。

重點提示 Key Notes

版本落差陷阱：若系統長期未升級，緊急補丁反而可能無法安裝，或安裝後造成應用程式崩潰。這是 CDMO 常見的技術債——省下的升級維護費用，最終在緊急時刻加倍奉還。

部分退役系統（partially decommissioned）：仍需保留查閱能力的唯讀系統，若原廠已停止支援，網路隔離是必要的風險降低措施，不是可選項。

多廠區系統：在一個廠址完整測試後，其他廠址可以減少測試——但這需要在策略文件中明確規定，並有標準化平台作為前提。

Appendix 8 — Outsourcing (p95–p99)

原文內容 Original Text

1. Definition of Responsibilities

Outsourcing involves the transfer of management and operations of a company's IT Infrastructure to an external company. Scope includes: operation of a data center and/or networks; operation of systems and/or processes (e.g., Help Desk); hardware/software component build (e.g., workstation).

The contracted service provider may have staff located on the regulated company's premises or provide services from a remote location (if on a different continent, 'offshoring' is sometimes used).

"The regulated company remains responsible for the regulatory compliance of their IT operations regardless of whether they choose to outsource/offshore some or their entire IT Infrastructure processes to external service provider(s). Compliance oversight and approvals cannot be delegated to the outsource partner."

The external service provider should focus on: regulatory education (all staff understand compliance impact); creation and maintenance of SOPs and work instructions; creation and maintenance of quality records; QA and QC of their IT processes.

2. Special Considerations

The location from which the outsourcing company provides services should be assessed. Services on-site should be managed per the company's standard security processes. Services from outside the regulated company's boundaries may require additional controls (e.g., for remote management of a data centre or a Help Desk).

3. Contracts

The regulated company is responsible for interpreting GxP regulations and defining appropriate requirements for the IT Infrastructure, provided to the external service provider in IT industry best practice terminology.

Key Performance Indicators (KPIs) and acceptance criteria should be defined by the regulated company. Contracts should handle ownership and retention of documents and records — if a contract is ended, the regulated company needs access to those documents or records at a later date.

Contracts should clearly state conditions under which the regulated company or regulators can undertake on-site audits to verify conformance. Expert support required on site to facilitate inspections, agreed notifications, and alert levels also should be specified.

4. Service Level Agreements

Where services are requested by one organization and provided by another, a formal SLA should be established. An SLA will usually be included as part of a contract and clearly state KPIs and associated fees.

A typical SLA will specify:

  • contacts on either side
  • duration of validity and circumstances triggering reviews
  • pre-requisites and customer deliverables or involvement
  • scope and nature of required services
  • metrics in the form of KPIs
  • records demonstrating fulfillment of specified service levels
  • pricing arrangements, including penalties for shortcomings
  • reports, scope, frequency, distribution
  • audit provisions, including readiness to facilitate inspections from regulatory authorities

5. Audits

Before entering into an outsourcing arrangement, a supplier audit should be considered. Staff from the regulated company's QA should be allowed to repeat initial audits at suitable intervals or if conditions warrant investigation.

The regulated company is responsible for authorizing appropriate resolution or mitigation of non-compliance areas arising from audits. The regulated company should include tracking of these within its CAPA program.

A non-disclosure agreement should be entered with the supplier to specify what information may and may not be shared.

6. Training Requirements

Regulations require that personnel have the appropriate combination of education, training, and experience. This applies to external supplier personnel as well. The service provider should ensure their staff is trained in regulatory compliance. Such training should be documented.

導師解析 Tutorial Commentary

重點提示：合規監督責任不可委外

這是本附錄最關鍵的原則。無論 IT 系統由哪家供應商管理，法規責任永遠在受管制公司身上。就像藥廠委外生產，受託方（CMO）執行製造，但合規責任仍在委託方（MAH）。

對 CDMO 的實際意涵：即便 IT 基礎設施完全委外給 AWS 或本地資服公司，FDA/EMA 稽查時，你仍需要能展示對供應商的有效監督——包括稽查紀錄、KPI 追蹤、CAPA 執行。「供應商說他們做了」不是合格答案。

比喻說明 Analogy

委外 IT 就像餐廳外包食材供應。廚師（受管制公司）對菜品安全負責，即使食材是供應商送來的。合約定義食材規格（KPI），定期稽查供應商的廚房（Supplier Audit），食材抵達時要驗貨（入廠查核）。若食材出問題，顧客告的是餐廳，不是供應商。

SLA 必要元素清單 SLA Essential Elements

合規相關必要項目（稽查重點）：
1. KPIs 與接受標準（量化績效指標）
2. 稽查權利條款（含法規機關現場稽查）
3. 記錄保存所有權（合約結束後仍可取得）
4. 事故通報等級與時效（Alert Levels）
5. 培訓要求（供應商人員需受 GxP 訓練）
6. 不合規時的處理程序與罰則

核心概念：記錄保存所有權

合約必須明確規定：若終止合約，受管制公司仍可存取所有 GxP 相關記錄。這在實務中常被忽略，直到供應商倒閉或合約到期才發現記錄被「鎖住」，無法配合稽查——這是嚴重的合規風險。

實務應用 Practical Application

CDMO 委外雲端基礎設施給 AWS / Azure 時，必須在合約中明確：

  • FDA 稽查人員能否要求現場查核雲端服務商的資料中心？（需寫入合約）
  • CDMO 離開雲端平台時，所有 GxP 記錄如何移轉？格式？時限？
  • 雲端供應商的安全事件通報義務（多少小時內必須告知客戶）？

這些都是合約談判時必須確認的項目，不是上線後再想。

Appendix 9 — Server Management (p99–p105)

原文內容 Original Text

1. Introduction

The objective of the Server Management process is to ensure that specified requirements for operational availability, performance, and security are consistently fulfilled by the server. An important part of the process is to manage server configuration and changes needed to meet the objectives.

All quantitative measures provided in this Appendix are derived from actual situations and are meant to assist companies, not to prescribe absolutes.

2. Backup and Restore

Failures of computer systems may result in a loss of data. The backup of electronic data is consequently essential to preserve the integrity and availability of the data.

2.1 Backup

There are two main types of backup:

  • Full backup: a copy of all data on a given system
  • Incremental/differential backup: a copy of data that has changed since the last backup cycle

The following minimum requirements are recommended:

  • Full backup: once per week, one day per week
  • Incremental or differential: all other days

Backups should be monitored to ensure successful completion:

  • review of backup logs to ensure successful completion
  • proactive assessments conducted regularly to ensure all required systems are included
  • backup media management

2.2 Verification

An extensive verification of the backup should be made when data is considered critical for the business, although the quantity of data may prevent full verification from being performed.

2.3 Schedules

The type of data should largely dictate backup schedules. Enterprise-level data actively in use can be considered to be changing each working day, with backup normally taking place overnight. Defining the schedule for global systems requires careful consideration, as access by users from other locations may create conflicts. Certain application data files cannot usually be backed up while in use — backup schedules may require augmented processes coordinating with deactivation of applications.

2.4 Restore

When a restore request is received, consideration should be given to whether the person requesting the restore is authorized, and that they are authorized to view the data being restored. When restoring data, checks should be made to ensure that more recent versions of files being restored are not overwritten, unless specifically requested. Consideration should be given to enable recovery to the point of failure.

Restoration beyond clearly defined single records or files may pose a challenge in assuring that all related records are brought back to the same state or point in time to retain overall integrity. This can be achieved only through carefully planned interactions with the system/data owner representatives.

2.5 Storage

A process for protecting backup data should be established to prevent loss, damage, and unauthorized access. Controls include:

  • secure storage facilities with proper access controls and environmental conditions
  • indexing and labeling capabilities and means to timely retrieval during inspections
  • detecting the end of retention requirements and notifying data owner
  • ensuring changes are carried out under change control
  • securely destroying data given necessary authorization

Although manufacturers give an expected lifetime for media, the expected lifetime when containing data is considerably less. Provisions should be made to refresh data after a defined period.

To ensure ability to recover data in a disaster, backup media should be stored at a secure 'off-site' facility — geographically diverse from the location where the system is housed.

2.6 Testing

The requirement to test the backup process should encompass any off-site storage solutions. The test should aim to restore data from historical backups, which may require handling of off-site backups. Restoring actual backups and comparing against the unchanged original file, where possible, gives the opportunity to check that the whole data path maintains data integrity. This periodic activity may certify both the backup and restore procedures.

導師解析 Tutorial Commentary

核心概念：全量 vs. 增量備份策略

Full Backup（全量備份）：每週一次，備份所有資料。優點：還原簡單。缺點：時間長、佔空間。

Incremental Backup（增量備份）：每日，只備份自上次備份後變更的資料。優點：快速、省空間。缺點：還原需要依序合併多個備份。

GxP 最低要求：全量每週一次 + 增量每日。這不是建議，是底線。

比喻說明 Analogy

備份策略就像工廠的照片記錄：每個月拍一次全廠大合照（全量備份），每天記錄今天哪裡有什麼變化（增量備份）。出事時，先找最近那張大合照，再依序套上之後每天的變化，就能還原到事故當天的狀態。

異地儲存就像把大合照的副本存在另一棟建築——同棟著火時，備份還能倖存。

重點提示 Key Notes

備份記錄的查核不等於測試還原：光看 backup log 顯示「成功」不夠。GxP 要求定期實際執行還原測試，並驗證還原後的資料與原始資料一致。「能備份」和「能還原」是兩件事。

還原授權管控：資料還原操作必須驗證申請人身份及其查看資料的授權。在 21 CFR Part 11 系統中，未授權還原等同於未授權資料存取。

媒體壽命陷阱：廠商標示的媒體壽命是空白媒體壽命，含資料的實際壽命遠較短。必須建立定期資料刷新（data refresh）計畫，避免長期備份資料自然損壞。

3. Technical Performance and Capacity Monitoring

The performance of devices and services need to be monitored to ensure that stated or implied expectations can be achieved. Review results or alarms will trigger maintenance, update, support, or disaster recovery activities.

Surveillance procedures should be developed and alert and action limits defined. For time-critical systems, the platform manager should ensure enough reserve resources to meet surges in demand.

Table A9.1 lists sample performance metrics for a process control server, as an aid in deciding metrics and limits in the context of all relevant deciding factors.

CPU load % >60 * >80 *
Server Work Queue # >2 >4
Available primary memory MB <50 * <10 *
Memory swaps Pages/sec >15 >20
Physical memory (disk space) % disk access time >60 >80
Physical memory queue length # >2 ** >4 **
Network interface kB total/sec >50 *** >80 ***

* Very much dependent on actual system set-up
** Depends on disk array configuration
*** Depends on application and network topology

Monitoring should take place on a daily basis using applicable software tools. A baseline should be created at the time of installation, and actual values should be considered based on those readings.

When using complex Process/System Management Applications, monitoring and event reaction could be automated.

4. Remote Management

When servers are remotely managed, e.g., via outsourcing or offshoring arrangements, it is important to maintain the required level of security. Consideration should be given to the use of secure communication channels.

導師解析 Tutorial Commentary

核心概念：Alert Limit vs. Action Limit

Alert Limit（警戒限）：系統開始發出警報，但尚未到達必須立即行動的程度。對應製造環境的環境監控警戒限——需要調查，評估是否趨近行動限。

Action Limit（行動限）：必須立即採取糾正措施，如重啟服務、擴充資源、啟動災難恢復程序。對應環境監控的行動限——超過即觸發 CAPA。

這個雙層限制結構（Alert → Action）與 GxP 製造中的 OOT / OOS 處理邏輯完全一致，對 CDMO 人員來說是熟悉的概念框架。

重點提示 Table A9.1 解讀

表中數值僅供參考（非絕對規定），應在系統安裝時建立基準線（Baseline），再依實際環境調整限制值。

對即時系統（Real-time system，如製程控制伺服器）而言，CPU >80% 或可用記憶體 <10 MB 是必須立即行動的警號——因為延遲回應直接影響製程控制，進而影響產品品質。

每日監控 + 自動化警報是合規要求，不是選擇。監控記錄本身就是 GxP 文件。

比喻說明 Analogy

伺服器效能監控就像無塵室的環境監控系統：設定警戒限和行動限，持續即時記錄，超標自動發警報，必須有 SOP 規定超標時的應對程序，且所有記錄都要保存以供稽查。兩者邏輯完全相同，只是監控對象從粒子計數換成了 CPU 使用率。

實務應用 Practical Application

CDMO 遠端管理伺服器（例如總部 IT 管理分散在各廠的系統）時，安全要求包括：

  • 使用加密通訊通道（如 VPN、SSH）——非加密的遠端存取是重大安全漏洞
  • 遠端存取本身也需要存取控制記錄（WHO、WHEN、WHAT）
  • 若透過委外供應商遠端管理，合約中必須明確規定安全通訊標準
