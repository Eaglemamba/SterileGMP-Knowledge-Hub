Appendix D6 — System Descriptions / 系統描述

D6: System Descriptions

26.1 Introduction

This appendix provides guidance on the contents of System Descriptions. EU GMP Annex 11 requires that there is an up-to-date description of critical GxP regulated computerized systems:

"For critical systems an up to date system description detailing the physical and logical arrangements, data flows and interfaces with other systems or processes, any hardware and software pre-requisites, and security measures should be available."

The need for a system description may be covered by one or more existing specifications or other documents, or a separate document may be produced. The principal use of such a document is to help new users and regulators understand what the system does, and as such is written in non-technical language as far as possible.

26.2 Scope

This appendix covers information that may be required for a wide range of GxP computerized systems. Not all the information will be required or relevant for all systems. The level and detail of information should be based on system risk, complexity, and novelty.

26.3 Guidance

26.3.1 General Guidelines

The system description should be maintained up-to-date throughout the life cycle of the system. For complex systems spanning multiple departments or sites (e.g., ERP systems), a separate document may be appropriate. For simpler systems it is common practice to include the system description in another specification or other document.

The development of the system description may begin early in the life cycle and evolve iteratively as the development process progresses. A complete system description meeting regulatory expectations should be established before the system is released for operational use.

The system description should be subject to change control and periodic review.

26.3.2 Contents of the Document

Listed below are topics that may be required in the system description depending on the nature and type of system. The guidance is neither prescriptive nor exhaustive. The system description should cover only the main features — detailed information on specific topics should be included in other specifications and not repeated.

26.3.2.1 Introduction

This should explain the context of the system within the business process and regulated company in general, considered from the following perspectives as appropriate:

  • Departmental
  • Site-wide
  • Division-wide
  • Global

26.3.2.2 Main System Functionality

A description of the key functions of the system, both GxP and non-GxP. The functions may be grouped to maintain the description at a high level. The use of diagrams is encouraged to explain relationships between key functions.

26.3.2.3 Regulatory Impact

This should include a description of the key GxP functions of the system. The impact that the system has upon patient safety, product quality, and data integrity should be considered. Other regulatory requirements should also be covered as appropriate.

26.3.2.4 The Computing Environment

This may be covered by a high-level diagram of the architecture supporting the system, covering as appropriate:

  • The infrastructure that supports the system (e.g., server configurations, storage arrangements)
  • Interfaces to users
  • Interfaces to equipment
  • Interfaces to other systems
  • Interfaces outside the company
  • The flow of data through the interface
  • Security features such as firewalls

26.3.2.5 System Components

An indication of the main hardware and software components should be provided. This may include servers and storage devices, operating systems, databases, and application layers, plus references to configuration documentation. A detailed inventory of all components is not required here.

26.3.2.6 System Interfaces

An overview of the key interfaces to other systems and equipment, and the data flowing to or from the system.

26.3.2.7 Access Control

An overview of the access control features of the system, both physical (if relevant) and logical.

26.3.2.8 Security Controls

An overview of the established system security controls both physical and logical, including software for the protection of data and records (e.g., virus protection software).

26.3.2.9 Electronic Records and Signatures

An indication of the types of electronic records created and managed by the system, and the type of electronic signatures used, should be provided if relevant.

26.3.2.10 Glossary

Definitions of any terms that may be unfamiliar to the readership should be provided.

D6 中文解析：系統描述

比喻：系統描述是GxP系統的「護照」

就像護照一樣，系統描述必須在系統「出關」（上線投入GxP使用）之前就備齊。它用非技術語言說明這個系統是誰、做什麼、跟誰連線、誰能進入。

法規查核官員（稽核員）第一眼看的往往就是這份文件——他們想快速理解這個系統的「邊界在哪裡」、「管到哪些資料」。護照沒更新，就是問題所在。

EU GMP Annex 11 的明確要求

Annex 11 明文規定「關鍵系統」必須有up-to-date的系統描述，涵蓋：

  • 實體與邏輯架構（Physical & Logical arrangements）
  • 資料流與介面（Data flows & interfaces）
  • 硬體/軟體先決條件（Prerequisites）
  • 安全措施（Security measures）

這不只是「最好有」，而是法規硬性要求。很多公司在稽核時才發現系統描述已過期三年，這是常見的483觀察項來源。

CDMO實務要點：什麼時候需要獨立的系統描述？

GAMP 5說複雜系統（如ERP、MES）才需要獨立文件；簡單系統可以把描述放進URS或其他規格文件中。

在CDMO環境，以下系統通常需要獨立的系統描述：

  • LIMS（Laboratory Information Management System）
  • MES（Manufacturing Execution System）
  • EBR/eBatch系統
  • 跨廠區共用的ERP模組

SCADA或獨立HMI通常可以將系統描述整合進IQ/OQ文件。

稽核現場重點：邊界定義

稽核員在查電腦化系統時，最重要的問題之一是：「這個系統的邊界在哪裡？」系統描述必須清楚回答：

  • 哪些資料在系統內部管理
  • 哪些介面連到外部系統
  • 誰負責哪個部分（系統擁有者 vs 供應商）

邊界不清楚 = 責任不清楚 = 稽核風險。

Change Control + Periodic Review 的重要性

系統描述必須受到變更管制並定期審查。常見的失效模式：系統已升版（例如從v8升到v9），但系統描述還停在舊版架構。這在稽核中會被認定為「documented system不符合實際運作狀態」。

建議：每次系統重大變更（infrastructure change、interface增減）必須觸發系統描述審查。

系統描述 vs. 其他驗證文件

系統描述是「門面文件」——它不重複其他規格的細節，但提供整合性的概覽。

Appendix D7 — Data Migration / 資料遷移

D7: Data Migration

27.1 Introduction

This appendix provides guidance for the migration of electronic data in the regulated environment, covering planning, execution, and reporting. Migration activities should be focused on aspects critical to patient safety, product quality, and data integrity, commensurate with risk, complexity, and novelty.

27.3 Guidance

27.3.1 General Guidelines

Data migration is the activity of transporting electronic data and associated metadata from one system to another, or the transition of data from one state to another. Data migration can occur often during and/or at the end of the life cycle of various computerized systems. Migration efforts can vary greatly in scope, complexity, and risk, including:

  • An in-place version upgrade of a database or application
  • Data conversion (e.g., from one supplier's database to another)
  • Same system – different platform migration (e.g., on-premise to cloud)
  • Migration from one source system to a different target system
  • Migration from multiple source systems to a single target system

The ultimate goal of any data migration is to have data that remains usable and retains its contextual meaning. Quality management controls should be in place to ensure data migration efforts are successful, compliant, and repeatable. Each data migration should be managed within the framework of a data migration plan and report, or through following an established data migration procedure.

27.3.2 Quality Management

27.3.2.1 System Life Cycle

Data migration may take place multiple times during the life cycle of a single computerized system, such as during initial system development and deployment, application upgrades, and system retirement. A data migration SOP may be the best method for describing and documenting the process expectations, including quality and compliance requirements. It must utilize a risk-based approach to ensure the effort is appropriately rightsized.

27.3.2.2 Risk Management

The following should be evaluated when migrating regulated data:

  • The inherent quality and compliance risk associated with the data being migrated, including impact upon patient safety, product quality, and data integrity; risk associated with the business processes; and risk associated with incomplete source data
  • Risk to the business due to systems being unavailable or data being unreliable
  • The level of complexity (e.g., multiple source or target systems; multiple phases; a high degree of data transformation)
  • Technology risk due to the use of complex or leading-edge systems or tools
  • Risks associated with the use of system vendors and third-party contractors

27.3.2.3 Configuration Management and Change Control

Migration of electronic data in the regulated environment should be performed under change control. All appropriate data migration project documents and tools should be controlled using configuration management and good documentation practices, including ALCOA+ principles.

During a data migration project, system changes unrelated to the migration should be prohibited, as unrelated system changes can increase complexity and overall project risk.

Typical good practice is to use one or more intermediate staging areas for data extracted from source systems prior to being loaded into the target system. At least three system platforms should remain under configuration management during the entire project: the source system(s), the staging area, and the target system. A rollback strategy should be established, including a full source-system backup prior to any migration activities.

27.3.3 Areas of Concern

27.3.3.1 Fitness of Software Tools for Intended Use

Data migration efforts typically involve software tools to automate extraction, transformation, loading, and verification activities. These tend to be GAMP software Category 1 infrastructure tools or Category 5 custom applications. The infrastructure tools and custom applications should be fit for intended use. For software tools that move or transform data, there are three principal areas of risk:

  • Data will be moved or transformed incorrectly or incompletely (incorrect data format, accuracy and/or completeness of the information, loss of information, orphan records)
  • Data residing in the target system will be harmed (deletion and/or overwriting of data, breaking and/or modification of data links, loss of metadata)
  • In cases where not all data is being migrated, residual data in the source system is adversely affected by the removal of the migrated data

Through the use of data verification, these risks can be significantly mitigated and therefore the rigor of software migration tool verification can be reduced. Development and approval of a data mapping table is still necessary.

27.3.3.2 Data Verification

Data should be verified each time it is moved or its state is transformed. There are two general types of post-migration data verification:

  • Test environment verification: A target test system is initially populated with data, then a migration test run is performed, and finally data is verified to show that all required data migrated successfully. A typical approach is to work with a relatively small amount of data, which can then be completely verified.
  • Operational environment verification: The same intent but the amount of data is typically very large. Automated software tools can be used to verify 100% of data in the target environment.

An important part is confirming that all of the required data has been migrated. Verification techniques such as checksum can be used to corroborate complete data transmission and locate failed records. Objective evidence of data verification should be generated.

27.3.3.3 Reliability of Source Data

The reliability of the source-system data must be considered as part of the migration process. If the reliability of the source system and data are unknown, the veracity of data migrated from the source system may not be verifiable; additionally, the migrated data will mix with reliable data already in the controlled target system. If inconsistencies are possible, they should be documented with justification of why the migrated data should be trusted.

27.3.3.4 Usability of Migrated Data in Target System

Four main issues to consider:

  • Target-system functionality does not allow performance of tasks previously carried out in the source system
  • Lack of completeness of migrated data affects the usability of the data
  • Separate migration of metadata or configuration may also be required (e.g., user groups and access rights)
  • Need for multiple migrations through various versions of a system to ensure full compatibility in the final target system

27.3.3.5 Audit Trails

Audit trails can be problematic for data migration efforts. If the target system has an audit trail but the source system does not, documentation should be created reflecting that auditing for migrated records began when they were loaded onto the target system. If both the source and target systems have audit trails and it is technically feasible, the audit trail should be migrated along with the audited data. If technically unfeasible, the original audit trail should be archived in a format that can be retrieved and used for support or investigational purposes.

When possible, a computer-generated audit trail should be created during the movement and transformation activities associated with the data migration effort. If the audit trail must be turned off during migration, the rationale should be documented.

27.3.3.6 Use of System Vendors and Third-Party Contractors

System vendors and third-party vendors should be assessed/audited to ensure they are capable of performing the work and that tools utilized are validated for their intended use. Master Service Agreements (MSAs), SLAs, Statements of Work (SOW), and/or quality agreements should be established.

27.3.4 Data Migration Plan

Every data migration project should have a data migration plan as a required deliverable. The data migration plan should describe the entire migration process, including as a minimum:

  • Migration project purpose and scope
  • System description(s)
  • Roles and responsibilities
  • Required deliverables
  • Risk-management strategy
  • Configuration management strategy (source, staging, and target environments)
  • Software tool overview and strategy for ensuring compliance and fitness for intended use
  • Migration steps, technical activities, and impacted business area activities
  • Data mapping and modeling activities
  • Transformation rules
  • Data verification strategy and acceptance standards
  • Cutover plan (including communication activities)
  • Rollback strategy

The data migration plan should be approved by the process owner, system owner, quality unit, and SMEs as appropriate.

27.3.5 Data Migration Report

The data migration report summarizes the activities conducted during the data migration effort. It describes any anomalies or deviations encountered and lists the results of verification activities. Because the report will be used to establish the reliability of the migrated data, it should clearly state the overall outcome of the migration activity (e.g., full success, partial success, failure). The data migration report should be approved by the same individuals who approved the migration plan.

D7 中文解析：資料遷移

比喻：資料遷移是「搬家」，不只是「搬箱子」

很多人以為資料遷移就是把資料庫的資料複製過去。但GAMP 5提醒我們：搬家時不只要搬家具（資料），還要確保：

  • 家具到了新家還能用（可用性）
  • 鑰匙、門禁也要跟著設定好（metadata、存取權限）
  • 搬家過程有完整紀錄（稽核軌跡）
  • 搬錯了能搬回來（rollback strategy）

CDMO最常踩的坑：從紙本或舊LIMS遷移到新系統

CDMO環境中最常見的資料遷移場景是：更換LIMS（舊系統 → 新系統），或將紙本批次記錄數位化後匯入EBR系統。

這種情況特別棘手，因為：

  • 舊系統的「資料可靠性」難以確認（有沒有audit trail？）
  • 紙本資料沒有audit trail，遷移後要怎麼交代稽核軌跡的起始點
  • 資料格式不統一（例如舊系統用文字儲存數值，新系統需要數字型別）
  • 業主覺得「copy-paste就好」，低估了驗證工作量

三環境架構：Source → Staging → Target

GAMP 5明確建議使用staging area（暫存環境）作為中間緩衝：

  • Source system：原始資料，在整個遷移期間不得有無關的系統變更
  • Staging area：資料清理、格式轉換、規則應用的工作區
  • Target system：最終生產環境，只有驗證通過的資料才進入

三個環境都要在configuration management控制下。Rollback計畫要在遷移開始前就準備好。

驗證策略：測試環境 vs 正式環境

測試環境驗證：先用少量資料（例如100筆）在test環境跑一次完整遷移流程，並進行100%比對驗證。目的是確認工具和流程可行，取得evidence before go-live。

正式環境驗證：資料量龐大，通常需要自動化工具進行100%比對（或統計抽樣加checksum）。自動化驗證工具本身的適用性也需要評估。

Checksum驗證的原理：計算來源資料的記錄數量與特定欄位的加總值，遷移後再次計算，兩個值相符才代表沒有漏筆或資料損失。

稽核軌跡的棘手問題

如果舊系統沒有audit trail，資料遷移到有audit trail的新系統後，稽核員會問：「這些記錄在遷移之前誰改過？」

GAMP 5的建議：在遷移計畫中明確記載「遷移前的稽核軌跡不存在」，並在每筆遷移記錄中標記migration date作為audit trail的起始點。若能在遷移過程中產生computer-generated log，更佳。

CDMO資料遷移計畫的最低清單

  • 遷移範疇與系統描述
  • 風險評估（source data reliability + tool risk）
  • Data mapping table（含轉換規則）
  • 三環境架構說明（source / staging / target）
  • 驗證策略（test run + production run）
  • Cutover計畫（誰負責通知使用者、系統停機窗口）
  • Rollback計畫（何時觸發、如何恢復）
  • 遷移報告範本（成功/失敗判準）
  • QA批准（流程擁有者、系統擁有者、QA、SME）

資料映射表（Data Mapping Table）：必備交付物

即使使用自動化遷移工具，也必須建立data mapping table，說明：

這份表格要放進遷移計畫，並獲得QA批准。

Appendix D8 — Agile Software Development / 敏捷軟體開發

D8: Agile Software Development

28.1 Introduction

Agile software development practices are widely adopted by many industries, including medical devices. Agile approaches focus on delivering quality and value to the customer at speed, and in an incremental fashion, enabling technical innovation and flexibility.

FDA's Technology Modernization Action Plan (TMAP) identified Agile software development and DevOps as foundational requirements for a modern FDA technology infrastructure. Such approaches are very well suited to the development of GxP regulated systems if correctly applied by trained and qualified practitioners supported by appropriate tools.

This appendix provides a summary of the principles underpinning Agile and illustrates how it can be implemented in a way that is aligned with GAMP 5 and GxP principles. The focus is on how to use well-implemented standard Agile processes to deliver software for GxP applications — it does not advocate modifying Agile for GxP (e.g., by superimposing linear V-model activities).

The planning, specification, verification and reporting activities within this Guide are not inherently linear, and the approach described is designed to be compatible with a wide range of models, methods, and schemes including incremental, iterative, and exploratory models.

The key principles behind Agile software development are that of discovery and of iteration (ongoing changes) as opposed to waterfall software development, where there is a linear flow of defining all requirements before transforming them into a complete set of specifications that are then configured/coded before testing commences.

Agile Manifesto Statements

"Individuals and interactions over processes and tools
Working software over comprehensive documentation
Customer collaboration over contract negotiation
Responding to change over following a plan"

These principles need to be taken in the context of emphasis rather than a binary choice.

28.2 Scope

Agile software practices can be applied across many types of computerized systems, including custom application development and configuration of configurable products in an incremental manner. Where the regulated company is looking at less clearly defined scope/requirements, Agile is likely to be the preferred approach to commence with a discovery phase to develop the initial backlog requirements and enable faster initial system deployment and subsequent incremental development.

28.3 Guidance

28.3.1 Agile Basics (Scrum Framework)

There are many different frameworks for Agile; the popular Scrum framework will be used as the reference. Agile is not a reason to accept a lower level of quality or GxP compliance. Evidence of fitness for intended use may be generated and maintained in tools and supporting systems, rather than in traditional documentation.

  • A Product Owner is responsible for collecting requirements into a product backlog, typically containing larger sets of requirements (epics) which are further split into sets of (user) Stories.
  • The Scrum cross-functional team takes a subset of the backlog items and develops/delivers and tests these during a sprint.
  • A sprint retrospective evaluates the business value of the completed sprint, and drives continuous improvement.
  • The cycle then repeats with another subset of the backlog.

Sprints are governed by time and designed to be a short period, 2 to 4 weeks. Backlog items unable to be completed during the Sprint return to the backlog.

Regulated company quality roles typically provide oversight and subject matter expertise on regulations and potential areas of product quality, patient safety, and regulated data impact. Ultimate accountability for quality and compliance lies with the regulated company.

The concepts of a Definition of Ready (DoR), Definition of Done (DoD), and also a Minimum Viable Product (MVP) are used to determine activities to be completed before a requirement can be considered complete and to define the minimum set of features needed for initial release.

Due to the iterative nature of Agile software development, regression testing, often performed using automated test tools, is typically applied to ensure stability of the system within sprints.

28.3.2 User Requirements

With Agile, by planning and storing requirements differently and following Agile principles, teams can successfully deliver effective and useful software in a controlled way that is compliant with GxP regulations, with a clearly identifiable set of fulfilled requirements at the point the system is released into the live GxP operational environment. The delivered description of system functionality is provided by the list of completed Agile artifacts such as epics and user stories obtainable directly from the Agile software development tools.

28.3.3 Mindset and Culture

A shift in culture and behaviors is required, according to the following principles:

  • An agile-first mindset and ways of working, using a Discovery Mindset, rather than a Certainty Mindset
  • A culture of innovation and continuous improvement
  • Empowered teams with the ability to deliver across geographies
  • A consistent and repeatable way of agile delivery
  • Much-improved alignment across all stakeholders

The discovery mindset also applies to business process owners and quality groups. With the traditional approach they are involved at the beginning (requirements) and end (acceptance testing). Agile is a continuous cycle of involvement and the resourcing and planning impact of this needs to be taken into account.

28.3.4 Tools Instead of Documents

Agile is typically used with sets of tools providing control over the product backlog, configuration, testing, and release activities enabling shorter cycle times and higher quality. From a quality and GxP compliance perspective, tools perform an essential role in demonstrating that the system is fit for intended use, functionality can be traced back to requirements, and testing completed.

Tools manage a variety of Agile activities such as user-story risk assessments, testing, and traceability. A backlog with continuously changing user stories is extremely difficult to manage on paper; a tool can provide an efficient and effective means to manage user stories throughout updates and changes. Reports may be generated from tools, for instance during regulatory inspections to provide evidence of status.

The tools used as part of Agile are covered within the scope of Appendix D9 and do not require computerized system validation, but should be subject to risk assessment, assessed for adequacy by appropriate SMEs, used by trained and qualified individuals, and appropriate controls applied according to intended use.

28.3.5 Approvals and Acceptance

There is a misconception that approval means sign, and by extension that there is the need to comply with the life-sciences regulatory requirements on electronic signatures (e.g., 21 CFR Part 11). This is not the case for software life cycle deliverables (except for SaMD or software embedded in a regulated medical device). Approval can be achieved by many other means such as status change, email, audit trails, and in many cases evidence of acceptance rather than an approval is sufficient.

Modern tool sets frequently have immutable change logs built into them, recording exactly who did what and when against each product backlog item, and can be configured to enforce security rights.

28.3.6 DevOps

DevOps is an extension of Agile where the development and maintenance activities are combined within a single team. In a DevOps environment, systems will undergo many changes/releases. This concern can be mitigated with the level of control and oversight of the process. However, there is still a level of risk/impact assessment required for higher-risk GxP functions undergoing change.

Continuous Integration (CI) and Continuous Deployment (CD) is an extension of DevOps where there is increased use/reliance on automated software analysis tools. DevOps provides additional controls such as:

  • No development before an accepted user story
  • Source code review and automated testing as part of the CI process
  • Ensuring close alignment and constant engagement with the product owner

28.4 Agile Approach to Quality

By following the agreed Agile process and associated ceremonies, a state of control can be demonstrated and quality built into the product. User stories can be assessed against INVEST criteria:

  • I – Independent
  • N – Negotiable
  • V – Valuable
  • E – Estimable
  • S – Small
  • T – Testable

A typical GxP regulated system has functional requirements that have GxP impact along with nonfunctional requirements (e.g., security) that need to be in place to support GxP regulations including data integrity. There must be a "potentially shippable product at the end of each sprint" — an untested product is not shippable. Agile testing uses a combination of exploratory testing and test automation with regression testing to verify that sprint developments do not impact correct functioning of software developed in previous sprints.

Agile is by its essence about tightly controlled management of change. Adherence to the defined Agile process (Scrum ceremonies such as sprint planning, daily meetings, sprint reviews, and sprint retrospectives) drives quality, ensures timelines can be adjusted quickly, and provides a focus on learning by addressing issues as they appear.

D8 中文解析：敏捷軟體開發

核心張力：Agile的速度 vs GxP的文件要求

Agile宣言說「Working software over comprehensive documentation」，但GxP說「沒有文件就等於沒發生」。這難道是根本矛盾嗎？

GAMP 5的回答是：No。關鍵在於「文件的形式可以改變」——工具裡的backlog記錄、test run的audit trail、sprint report，這些都是合法的GxP evidence。重點是可追溯性和完整性，而不是紙本或PDF的數量。

CDMO如何落地：Sprint文件策略

在CDMO的電腦驗證專案（例如新LIMS、MES）中採用Agile，建議的文件策略：

  • Sprint Planning：產出backlog item list（含GxP影響評估）→ 等同於傳統的功能規格片段
  • Sprint Execution：在工具中記錄user story完成狀態、test results、defect resolution
  • Sprint Review：產出sprint summary report（誰測了什麼、通過/失敗、未解決的問題）→ 等同於測試記錄
  • Sprint Retrospective：流程改善記錄 → 持續品質改善的evidence
  • Release：所有sprint的cumulative backlog + DoD verification = 驗證總結報告的基礎

Definition of Done (DoD) 與 GxP合規

DoD是Agile中最重要的品質控制機制。在GxP系統中，DoD應包含：

  • 功能已開發完成（code complete）
  • 單元測試通過（unit test pass rate > X%）
  • Regression testing通過（不影響之前sprint的功能）
  • GxP影響評估已完成（如有資料完整性影響，已記錄）
  • Defect已close或risk-accepted（附justification）
  • User acceptance已確認（業主簽核或工具狀態記錄）

一個未通過DoD的user story不能計為「完成」，也不能進入下一個sprint作為基礎。

INVEST標準：寫好User Story的關鍵

如果user story寫得不好（太大、太模糊、無法測試），整個Agile流程都會出問題。INVEST是一個自我檢查框架：

  • I (Independent)：這個story可以獨立開發，不依賴未完成的其他story
  • N (Negotiable)：細節可以討論，不是不可更改的契約
  • V (Valuable)：對使用者或業務有明確價值
  • E (Estimable)：團隊能夠估算開發工作量
  • S (Small)：一個sprint內可完成（通常1-3天開發量）
  • T (Testable)：有清楚的acceptance criteria，知道怎麼測試

電子簽名的誤解：Agile中的approval不等於21 CFR Part 11

很多GxP從業者看到Agile工具的「approve」按鈕就緊張：這需要符合21 CFR Part 11嗎？

GAMP 5明確回答：不需要（除非是SaMD或醫療器材嵌入軟體）。軟體生命週期文件（user story approval、sprint sign-off）的「批准」不等於法規上需要具法律效力的電子簽名。工具的audit trail（誰、何時、做了什麼）就足夠作為acceptance evidence。

DevOps + CI/CD 在GxP環境的適用性

DevOps將開發與維運整合到同一個團隊，CI/CD實現自動化測試和部署。這在GxP環境聽起來很可怕（「系統每天都在變？」），但實際上可行的前提是：

  • 每次change都有risk assessment（高風險GxP功能需要更嚴格的評估）
  • CI pipeline包含自動化regression testing
  • 部署需要DoD + change control gate
  • Compliance accountability在DevOps團隊內部，不是外部的人在審批

在CDMO中，這種模式適合SaaS-based系統（如cloud LIMS）的持續更新管理。

Scrum在GxP環境的角色對應

注意：QA在Agile環境中的角色是「嵌入式夥伴」，不是最後的守門員。這需要QA思維轉型。

Appendix D9 — Software Tools / 軟體工具

D9: Software Tools

29.1 Introduction

This appendix describes the recommended risk-based approach and considerations when using tools supporting computerized systems life cycle processes, IT processes, and IT infrastructure processes. Such tools do not directly support GxP regulated business processes or GxP records and data directly supporting the regulated product life cycle.

The tools described in this appendix are GAMP software Category 1. Classification as GAMP software Category 3–5 applies only to business applications (either GxP or non-GxP). Tools, like systems supporting IT processes and infrastructure, are all GAMP software Category 1 and should be managed through the application of good IT practices, within a defined IT management framework such as ITIL.

Such tools and systems supporting system life cycles, IT processes, and infrastructure should not be subject to specific validation, but rather managed by routine company assessment and assurance practices and good IT practices. Compliance and regulatory resources should not be focused on audit and review of activities related to systems with little or no direct impact to patient safety or product quality.

EU GMP Annex 11 requires that automated testing tools and test environments should have documented assessments for their adequacy, supported by appropriate controls following good IT practices.

29.2 Scope

This appendix applies to software tools used as part of software development, support, and maintenance activities. It does not apply to software components/applications that are part of a GxP regulated business process, or can directly impact GxP data/records directly supporting the product life cycle; in this case risk-based computerized system validation is required.

For example: software used to migrate or convert GxP records would require computerized system validation; however, a tool used to mask or obfuscate data to be used as system test data would not.

Test tools perform an important role by helping to assure thorough testing. Such tools also often store the associated test results, but even though these records may support the computerized system life cycle, the test tools themselves do not require validation. The records they maintain should be preserved and protected, and the tools need to work reliably — this can be achieved through a basic risk assessment and application of normal IT practices.

29.3 Guidance

29.3.1 Selection of Tools

Some basic assessments should be performed as part of tool selection. The key criteria are whether the tool is the appropriate tool to support the life cycle process we are trying to automate/support: does it meet the need of the SMEs who are going to use the tool, and is it fit for purpose? It is not envisaged that any tool would need to have anything other than a desktop assessment, unless the use was considered business critical.

29.3.2 Risk Assessment

Some of the factors to consider as part of risk assessment are:

  • Is the tool an industry standard one in common use?
  • What data/records are generated/stored by the tool, and does the tool have adequate controls to maintain data integrity?
  • What is the required operational lifetime of the tool, and how will this be supported?
  • Is the tool open source; are the download locations verified to avoid cybersecurity issues?
  • Will the tool require any level of workflow/configuration, including any key activities that need to be manually performed?
  • For automated tools, how are any alerts monitored and acted upon?
  • If the tool fails, could it impact an operational computerized system or infrastructure operation?
  • Could the tool have any potential negative impact on system or network performance?
  • Could the tool introduce any cybersecurity risks, e.g., facilitate the introduction of malware?

29.3.3 Inventory/Deployment

As part of good practice, it is sensible to have some form of inventory, for example within a Configuration Management Database (CMDB), of what tools are used by the organization, where they are deployed, and the nature of any data/records potentially supporting GxP and required retention period. Identifying a single point of contact/owner for the tools is also recommended.

29.3.4 Installation and Configuration

Tools, even when off-the-shelf, often will require some degree of configuration or parameterization in order to correctly install and establish any workflows and associated aspects such as naming conventions or access levels. These are basic good software engineering practices that need to be applied along with version and appropriate change control.

29.3.5 Life Cycle Management

Tools typically have a defined lifetime, upgrades, and eventually are retired and/or decommissioned. One person or role should be designated to be the owner of the tool to provide the required level of oversight. In general, tools that have the ability to impact system or network performance require more oversight than those used as part of software development activities. It is best practice to ensure tools in use are at versions supported by the supplier.

29.3.6 Security Considerations

Cybersecurity considerations should be included for tools. Tools that are deployed across the network (e.g., for performance monitoring) may require more detailed security analysis. Security considerations should be included as part of the initial risk assessment and also as part of the life cycle management (e.g., ensuring tools are maintained as patched and up-to-date). Access privileges for tools should be set as low as possible to allow the tool to perform the task.

29.3.7 Data and Records

Where tools are storing information/records that support the validated status of the system (e.g., system requirements, design, code, or test results), controls for maintaining integrity, security, and availability should be determined and applied. Note that increasingly tools and the records/data within them may be cloud-based, and therefore assuring data integrity and access throughout the retention period should take account of this.

The full set of data integrity controls that would be applied to primary GxP records are not required, as these tools do not maintain GxP regulated records. Most tools do support technical and transactional logging supporting integrity and investigations.

29.3.8 Approvals

For tools within the scope of this appendix, any approvals or acceptances associated with the tool are not subject to regulated electronic signature requirements. Where an approval or acceptance is required, there needs to be traceability to the person approving or accepting, and approval roles need to be defined.

29.3.9 Tools over Documentation

There may be a temptation, particularly where tools store test execution records, to export information into controlled GxP documentation when there is no quality benefit in doing so. This approach is not recommended — it is inefficient and there is the risk that documentation becomes out of step with the information/records within the tools. Tools should have sufficient controls to preserve the data/records.

There is the potential for regulatory inspections to request access to information in tools (e.g., system requirements, test results). Process owners need to be aware of what information relating to their computerized system life cycle activities is managed within tools. It may be possible to pre-configure reports to extract test results relating to areas of GxP functionality.

Tools and systems supporting life cycles, IT processes, and infrastructure (rather than directly supporting product life cycle processes) are not themselves GxP regulated systems and should not be subject to specific validation, but managed by routine company assessment and assurance practices and good IT practices.

D9 中文解析：軟體工具

關鍵問題：Excel macro需要驗證嗎？

這是CDMO環境中最常遇到的問題之一。答案取決於Excel的用途：

  • 用Excel執行GxP計算（例如OOS計算、環境監控趨勢分析）：Excel本身是工具（Category 1），但這個特定的macro/workbook是在執行GxP業務流程 → 需要CSV（Computerized System Validation）
  • 用Excel管理驗證測試的test scripts（不含GxP記錄）：這是開發工具用途 → 不需要validation，但需要基本risk assessment + change control
  • 用Excel自動生成validation protocol的模板：生命週期工具 → Category 1，不需要validation

關鍵判斷標準：工具輸出的記錄是不是直接支援product life cycle的GxP記錄？

EU GMP Annex 11的工具要求

Annex 11明確要求：automated testing tools和test environments必須有documented adequacy assessment。這不等於要validation，但必須有書面評估說明這個工具夠不夠好、是否適合其用途。

實務上的最低要求：

  • 工具識別與用途描述
  • Adequacy assessment（SME判斷：工具是否適合intended use）
  • 使用者訓練記錄
  • 基本access control
  • 納入CMDB（配置管理資料庫）

工具風險評估的關鍵考量

GAMP 5提供了九個風險評估考量點。在CDMO環境，最重要的是：

  • 工具儲存的記錄性質：如果工具存有test results或system requirements，這些記錄雖非GxP records，但支援validated status → 需要integrity controls
  • 雲端工具的資料保留：如果工具是SaaS（例如雲端Jira、GitHub Enterprise），須確保記錄在retention period內可取用
  • 工具失效的影響：若工具當機，是否影響已上線的GxP系統？（例如監控工具失效可能影響環境監控的警報）
  • 資安風險：開源工具的下載來源是否可信？工具是否可能引入惡意軟體？

「工具記錄不要複製進GxP文件」的重要原則

很多驗證工程師習慣把Jira/TestRail的測試結果截圖後貼進Word文件再上傳到文件管理系統。GAMP 5說：這樣做沒有品質效益，而且有風險——工具裡的記錄和文件裡的截圖可能不同步（工具裡的defect已close，但文件還沒更新）。

更好的做法：在驗證報告中引用工具的report（例如：「詳細測試結果見TestRail Report #TR-2026-042」），並確保稽核前能展示工具的原始記錄。

稽核應對：檢查官要求看工具記錄

GAMP 5提醒：法規檢查官可能要求看工具裡的記錄（例如system requirements、test results）。CDMO需要事先準備：

  • 知道哪些工具存有哪些系統的life cycle記錄
  • 能夠快速從工具中產生特定系統的GxP功能測試結果報告
  • 如果工具是第三方SaaS，確認在audit期間能取得資料存取權
  • 工具的access log能顯示誰在何時存取了哪些記錄

不要等稽核當天才發現「那個tool的管理員已離職，帳號停用了」。

工具生命週期管理：被忽視的維護義務

工具需要有designated owner（不是IT全負責，也不是驗證團隊全負責，要有明確的single point of contact）。工具的退役計畫同樣重要：

  • 工具退役前，確認其中儲存的記錄已轉移或存檔
  • 確保記錄在regulatory retention period內（通常10年以上）仍可讀取
  • 版本升級需要change control + 確認不影響既有記錄

Cloud-based工具特別要注意：vendor若停止服務，你的test records在哪裡？

GAMP Category 1：工具 vs 業務應用的分野

Section Summary — 本節總結

Appendices D6–D9 of GAMP 5 (2nd Edition) address four practical topics that bridge the gap between theoretical GxP compliance requirements and modern software engineering practice.

  • D6 System Descriptions — A regulatory-facing document required by EU GMP Annex 11 for critical systems. It serves as the "passport" of a computerized system, describing its purpose, architecture, interfaces, access controls, and electronic records in non-technical language. Must be subject to change control and periodic review; must be complete before system goes live.
  • D7 Data Migration — One of the most underestimated validation activities. GAMP 5 requires a formal data migration plan (including data mapping, staging architecture, verification strategy, rollback plan) and report approved by QA, process owner, and SMEs. Three-environment staging (source → staging → target) under configuration management is standard good practice. Source data reliability, audit trail continuity, and tool fitness for use are key risk areas.
  • D8 Agile Software Development — Agile (Scrum-based) development is fully compatible with GxP compliance when correctly applied. Evidence of compliance lives in tools (backlog, test management, code repository) rather than traditional document stacks. Key governance mechanisms: Definition of Done, sprint testing (including regression), sprint ceremonies, and INVEST-quality user stories. DevOps/CI/CD adds further automation and control. Quality is built in throughout the process, not bolted on at the end.
  • D9 Software Tools — Development tools (backlog managers, test runners, code repositories, monitoring tools) are GAMP Category 1 and do not require computerized system validation. However, they need risk assessment, adequacy evaluation (EU GMP Annex 11 requirement), designated owners, CMDB inventory, and appropriate controls for any records they hold. The key decision point: does the tool's output constitute a primary GxP record directly supporting the product life cycle? If yes → validation required. If no → risk-based IT good practice suffices.

核心思維：GxP合規不在於文件數量，而在於可追溯性、完整性、以及整個生命週期中維持受控狀態（state of control）。Agile和現代工具鏈恰恰能夠以更有效率的方式實現這些目標——前提是要有正確的mindset和適當的治理機制。
