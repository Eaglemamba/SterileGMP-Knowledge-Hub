Appendix D4 — Management, Development, and Review of Software (軟體管理、開發與審查)

D4.1 Introduction & Scope

This appendix provides guidance for the management, development, and review of software. It also includes general guidance on coding practices. Software tools to assist with these activities are available, and in general, automation of these processes will lead to better compliance, efficiency, and reproducibility when compared to manual approaches.

Software development should be performed within a defined life cycle. The criteria for releasing developed and tested software should be understood and defined. Software released into the GxP environment should be based on, and meet, defined requirements.

This appendix is not intended to constrain the choice of development methods and models. Suppliers should select and use the most appropriate methods and models for their use. Any examples used, or technologies assumed, are intended to illustrate general principles and are not intended to be restrictive.

Scope: While the main focus is on software development using high-level and low-level languages, the principles of this appendix also may be applied to various techniques used for process control system software development/configuration.

Changes from GAMP 5 First Edition

This appendix has been updated in line with changes made to Appendix D1 (Specifying Requirements), and also to align with information in new appendices D8 (Agile) and D9 (Tools).

中文解說：D4 引言與範圍

D4 的定位：給誰看的？

D4 主要面向軟體供應商（如 LIMS、MES、ERP 廠商）和客製化開發團隊（自行開發 Category 5 軟體的製藥公司）。

核心訊息：GxP 環境中的軟體開發不是「寫完能跑就好」，而必須有定義的生命週期、版本控制、代碼審查，且須在發布前通過定義的驗收準則。

類比理解：軟體開發就像批次製造

製藥生產中，每批原料都有批號、配方、製造步驟記錄、QC 檢驗，且須符合規格才能放行。

軟體開發也一樣：每個版本 (build) 都應有版本號、需求對應、代碼審查記錄、測試報告，且須符合定義的驗收標準才能部署到 GxP 環境。

實務重點

D4 明確指出本指引不限定開發方法（瀑布、Agile 都接受），但要求生命週期必須被定義。這是 GAMP 5 的核心哲學：規定「要做什麼」，不規定「怎麼做」。

D4.3.1 Software Development

The software development method should be defined and regardless of the model used, should also define:

  • Coding standards
  • Code repository standards
  • Software development tools
  • Naming conventions
  • Configuration and use of build and deployment systems

Source code should be subject to review with the approach and extent of such reviews based upon the coding language, tools, and risk to patient safety, product quality, and data integrity.

D4.3.1.1 General Design and Documentation Principles

Software design artifacts should be sufficient to initially develop and subsequently support working software. The degree and nature of the design will be based on the life cycle model used, software complexity, code/configuration language, and interfaces. Design artifacts may be in documentation form or as records within software development tools.

It is good practice to include a Software Bill of Materials (SBOM) that provides the list of components/modules that make up the overall software and can also be used to identify/track inter-dependencies.

Software should ultimately:

  • Meet requirements
  • Be reliable and robust
  • Be maintainable
  • Be capable of handling error conditions
  • Be well designed and sufficiently commented

D4.3.1.2 Software Module Identification

Each software module should include the following attributes:

  • Module name
  • Reference/traceability to associated design artifacts
  • Constituent source file names
  • Module version number
  • Project name/reference (if applicable)
  • Brief description of the module
  • References/dependencies to other software modules
  • Any specific command files required to compile and build the module

中文解說：軟體開發生命週期

SBOM（軟體物料清單）— 新增亮點

GAMP 5 第二版新增了 Software Bill of Materials (SBOM) 的概念。這是仿照製造業 BOM（物料清單）的做法：

  • 列出系統由哪些元件、模組、第三方函式庫組成
  • 識別各元件間的相依關係
  • 在資安漏洞出現時，能快速判斷哪些系統受影響

在驗證 LIMS 時，要求供應商提供 SBOM 已成為新的最佳實踐。

軟體模組識別 = 藥品成分標示

就像每種藥品必須標明活性成分、賦形劑、批號、有效期；每個軟體模組也必須有版本號、對應的需求文件、依賴的其他模組。沒有這些資訊，維護和故障排查將極為困難。

稽核重點

FDA/EMA 查廠時，常見問題是：「你們的 LIMS 版本 3.2.1 與 3.1.0 有什麼差異？是否有變更記錄？」

若無法回答，代表版本控制不足，可能引發 483 observation。

D4.3.1.3–1.7 Version Control, Maintainability & Code Quality

Software Module Change Traceability

All software module changes should be clearly identified according to the chosen software development life cycle and technology used. It should be possible to obtain a history of changes made, which should include the following information for each change:

  • Change request number(s) (if applicable)
  • Applicable version number(s)
  • Dates change(s) made
  • Identity of person(s) changing the code
  • Summary of the changes

D4.3.1.4 Maintainability

Software should be written and structured in a way that a competent person (other than the original author) would be able to understand and modify it safely. The software should be created using the coding standards relevant to the technology.

D4.3.1.5 Open-Source Modules

Where modules are imported from an open-source code repository, updates as and when the open-source code is updated need to be determined. In addition, occasionally open-source modules may be retired, in which case future vulnerabilities will not be addressed, which increases risk.

D4.3.1.6 Removal of Dead Code

Dead code is code that cannot be executed due to the logic of the program, and should be removed. It is usually a symptom of poor maintenance. Code that has been included for purposes of testing or for later diagnosis during support work, and which can be configured on or off, is not regarded as dead code.

D4.3.1.7 Reliability and Error Recovery

"Defensive" coding techniques should be employed. Input values should be checked, and subroutines should check that the values of the received parameters are within range. In the event of a failure, a safe, documented error recovery or a controlled shutdown should be performed, with an informative error message.

Software Configuration Management

Software should be subject to a defined software configuration management process that includes version control of software code, scripts, and any relevant artifacts required by the technologies in use. Such change and version control is especially important during software development, as mistakes in this regard can lead to problems such as reintroduction of previously removed defects.

中文解說：版本控制與代碼品質

版本控制 (Version Control) 的 GxP 意義

版本控制不只是 IT 工具（如 Git），在 GxP 語境下，它必須確保：

  • 可追溯性：每次代碼變更都能追溯到對應的變更請求 (Change Request)
  • 不可篡改：已審查、已測試的版本不能被未授權修改
  • 可重現：能重建任何歷史版本，以便調查問題

Dead Code = 過期庫存

想像工廠倉庫中有一批過期原料，沒人知道能不能用。Dead code 也一樣——存在的代碼應該有明確目的。無法執行的代碼意味著維護不佳，可能有資安或數據完整性風險。

開源軟體 (OSS) 的特殊考量

GAMP 5 第二版顯著擴展了開源軟體的指引。許多 LIMS、MES 都使用開源元件（如 PostgreSQL、Python 函式庫）。監管重點：

  • 開源社群是否仍活躍維護？
  • 安全漏洞更新是否被追蹤？
  • 版本是否被鎖定並文件化？

防禦性程式設計 (Defensive Coding)

對 QC 系統特別重要。例如：儀器傳入異常值（如 -999 的 pH 值）時，系統應拒絕並記錄錯誤，而非接受並計算出荒謬結果。這直接關係到數據完整性。

D4.3.2 Source Code Review

Source code reviews have three objectives:

  • To ensure that programming standards, such as those described above, are consistently and correctly applied
  • To ensure that the code is written in accordance with the design
  • To identify and enable removal of defects

The review aims to ensure that the code is fit to release, and that the code can be effectively and efficiently maintained during the period of use of the application.

The review should be performed by at least one independent person with sufficient knowledge and expertise, in conjunction with the author of the code.

Automated source code review comparison tools are particularly useful when making small changes to existing code.

D4.3.2.1 Static Analysis of Software

"Static analysis involves a set of methods, supported by tools, used to analyze software source code or object code to determine how the software functions and establish criteria to check its correctness."

These include:

  • Control Analysis – analysis of the controls used in calling structure, control flow, and state transitions
  • Data Analysis – ensures proper operation is applied to data objects and data structures
  • Fault/Failure Analysis – identifies incorrectly specified or built components and incorrect behavior of components
  • Interface Analysis – checks the accuracy of interface structure, and the ability to prevent errors during user interaction

Formal Verification methods may also be considered for high-risk areas of code using formal mathematical methods to provide "proof" of algorithms under multiple input conditions.

A decision based upon known risks should determine the extent of software review required for a given project. The criteria for making this assessment, and the outcome, should be documented.

中文解說：源代碼審查

代碼審查的三個 GxP 目標

代碼審查 (Code Review) 在 GxP 環境中不只是品質工具，更是驗證活動的一部分：

  • 合規性確認：編碼標準是否被遵守？
  • 設計一致性：代碼是否實現了設計規格 (DS) 中定義的邏輯？
  • 缺陷識別：在測試前發現問題，成本最低

靜態分析 = 製程前的原料檢驗

靜態分析 (Static Analysis) 是在「不執行代碼」的情況下分析代碼結構。就像進廠原料先做 CoA 審查和鑑別測試，不需要先放入製程才發現問題。

工具範例：SonarQube（代碼品質）、Checkmarx（資安漏洞）

獨立審查 (Independent Review) 的重要性

GAMP 5 要求由不是原始作者的人執行代碼審查。這與 GMP 中「操作者不能是審查者」的原則一致。

在 Category 5 自開發系統驗證中，代碼審查的記錄會被監管機構查核。

D4.3.3 Third-Party Software (including Open-Source)

D4.3.3.1 Purchased Software

In most cases evidence of software control can be obtained through standard mechanisms for supplier assessment, such as supplier audit (see Appendix M2). Conclusions of the assessment should be documented. If inadequacies are noted, the regulated company may require remedial actions by the supplier, possibly enforced via a Service Level Agreement (SLA), or may decide that further testing is required.

D4.3.3.2 Free and Open-Source Software (OSS)

OSS often plays an important role in the infrastructure of regulated companies. Examples include the Linux operating system, R environment for statistical computing, and many web design, programming, and support tools.

Note that commercial software may contain open-source components incorporated into it. It is becoming more common to request a SBOM when evaluating new commercial software or validating in-house developed systems.

If a decision is taken to use OSS, then such software should be implemented in accordance with the appropriate life cycle activities as for commercial software, with consideration of the following:

  • Knowing the size and sustainability of the open-source community
  • Checking for development standards and good documentation
  • Knowing what version is in use — verify download from a reputable source and ensure code was not altered (via checksum)
  • Understanding the governance model for updates and patches
  • Keeping up-to-date with patches and updates; vulnerabilities are often exposed in software using outdated versions of open-source libraries
  • Participating in the open-source community

To determine whether there is an adequate level of software maturity and control, a risk assessment should be performed to determine the business and GxP risks of using OSS products, and to identify appropriate controls.

Appendix D5 — Testing of Computerized Systems (電腦化系統測試)

D5.1 Introduction

This appendix covers the testing of GxP computerized systems. It is not intended to offer a prescriptive approach — it does not require that testing must be done a certain way in a particular sequence. Instead, this appendix identifies the influences on testing and offers information on the types of testing and testing approaches that can be used selectively and in combination for optimal assurance that the system is fit for its intended use.

The application of critical thinking is vital to achieve effective testing that adds value and fulfills objectives such as:

  • Identifying defects so they can be corrected or removed before operational use
  • Reducing the risk of failures that might affect patient safety, product quality, or data integrity
  • Demonstrating that the system as installed and configured meets its requirements
  • Providing evidence that the system is fit for its intended use
  • Offering a basis for user acceptance
  • Meeting a key regulatory requirement
  • Ensuring that risk management controls are effective

Testing should be carried out in accordance with an appropriate test strategy based on the risk to patient safety, product quality, and data integrity. Testing may occur both in the vendor's development life cycle and in the regulated company's implementation life cycle.

Key Changes from GAMP 5 First Edition

  • Critical thinking should be applied when planning testing efforts such that the level of effort is commensurate to the risk acceptable within the organization
  • Testing by any means and in any part of the life cycle all contributes to finding defects and confirming fitness for intended use
  • Testing should not be limited to detailed and prescriptive step-by-step scripted protocols. Exploratory testing and other unscripted techniques are encouraged
  • Modern approaches may rely on records and artifacts in automated tools in place of formal specification and test documentation — either approach is acceptable

中文解說：D5 測試附錄引言

GAMP 5 第二版的測試哲學轉變

與第一版相比，第二版最大的改變是：

  • 去除處方化：不再要求一定要有 URS → FS → DS → IQ → OQ → PQ 的線性流程
  • 擁抱非腳本式測試：探索性測試 (Exploratory Testing) 獲得官方認可
  • 承認工具記錄的有效性：Jira tickets、CI/CD 日誌等工具記錄可作為測試證據
  • 批判性思維是核心：比規程遵從更重要

測試的七大目的 — 記憶要點

這七個目的覆蓋了測試的三個維度：

  • 技術維度：找缺陷、確認需求滿足
  • 監管維度：符合法規要求、提供符合使用目的的證據
  • 風險維度：降低影響病患安全/產品品質/數據完整性的風險、確認風險控制有效

The regulated company should define roles and responsibilities covering testing. Such definitions typically are included in an appropriate planning document, test strategy, or company procedure. Traditional roles and responsibilities may include:

The quality of the test cases and the functionality they cover directly impact the effectiveness of the testing. No matter who is creating the test case, they should be an SME in the associated field.

中文解說：測試角色責任

獨立性原則 (Independence Principle)

GAMP 5 強調兩個關鍵的獨立性要求：

  • 開發者 ≠ 測試者：寫代碼的人不能測自己的代碼（因為只會確認預期行為，發現不了設計盲點）
  • 執行者 ≠ 審查者：執行測試的人不能審查自己的測試結果

這與 GMP 中「雙人復核」原則相同，確保客觀性。

SME 在測試設計中的角色

GAMP 5 特別強調：測試案例的設計者必須是該領域的 SME。

以 LIMS 的樣品接收功能為例：

  • IT 測試師寫的測試：「填欄位、存檔、確認無錯誤」
  • QC SME 寫的測試：「測試樣品ID重複時系統是否警示」、「測試緊急樣品優先級標記」、「測試跨部門接收時通知邏輯」

SME 測試覆蓋業務場景，IT 測試覆蓋系統功能——兩者缺一不可。

供應商責任 vs 受規管公司責任

無論外包多少，法規合規的最終責任永遠在受規管公司。供應商的測試可以被利用（leveraged），但不能取代受規管公司的驗證決策。

中文解說：測試活動的影響因素

風險為基礎的測試策略 — 核心概念

GAMP 5 最重要的訊息之一：不是所有功能都需要相同程度的測試。

測試投入應與風險相稱 (commensurate with risk)：

  • 高風險功能（如電子簽名、批次放行計算）→ 詳細腳本式測試 (Scripted Testing: Robust)
  • 中等風險功能（如報表生成）→ 腳本式有限測試 (Scripted Testing: Limited)
  • 低風險功能（如 UI 偏好設定）→ 非腳本式探索性測試即可

LIMS 驗證的測試規劃實例

以驗證一套 LIMS 為例，根據 Table 25.1 的邏輯：

  • 需求定義：先完成 URS/FRS，確認 GxP 影響範圍
  • 供應商評估：供應商有 GAMP Category 4 軟體的完整測試記錄？可降低受規管公司的測試負擔
  • 功能風險評估：樣品追蹤功能（高風險）vs 報表格式設定（低風險）
  • 測試規劃：高風險功能需 100% 腳本式測試；低風險功能可用探索性測試

驗證報告 (Validation Report) 的關鍵內容

驗證報告不只是「所有測試通過」的總結，還需要：

  • 記錄測試記錄的存放位置（工具中？紙本協議書？）
  • 列出未解決的偏差及其影響評估
  • 確認驗證策略是否按計畫執行

D5.9.1.1 Installation Qualification (IQ) Context for Computerized Systems

Although the term is not used in this Guide (which refers to "installation verification activities" in general), many companies still use the traditional process validation term Installation Qualification (IQ).

The purpose is to verify and document that system components are combined and installed in accordance with specifications, supplier documentation, and local and global requirements. Installation testing provides a verified configuration baseline for subsequent verification and validation activities, and verifies any installation methods, tools, or scripts used.

For cloud-native or virtualized software, the use of automated installers combined with configuration management tools and service management tools will be more effective than conventional installation approaches. If they provide dashboard functionality, then a review-by-exception approach can replace many of the static verification activities.

Whatever the approach, the availability of appropriate technical information should be verified. Examples include:

  • User and technical guides; SOPs; Training schedules
  • SLAs; Security procedures; Log books
  • Hardware inventory; Instrument lists; Specification sheets
  • Certificates and calibration procedures
  • Software inventory (including installation procedure, system software list, application software list, data list, initial data settings)
  • Preventive maintenance program; List of critical spare parts

中文解說：IQ/OQ/PQ 在電腦化系統中的意義

重要：電腦化系統的 IQ/OQ/PQ 與設備確認不同！

許多工程師和 QA 用設備確認的思維套用到軟體驗證，這是常見錯誤。讓我們用 LIMS 為例比較：

LIMS 系統 IQ：安裝確認

目的：確認系統已按規格正確安裝

檢查項目：

  • 硬體規格確認（CPU、RAM、儲存空間是否符合最低需求）
  • 作業系統版本、補丁級別確認
  • LIMS 軟體版本確認（與採購規格一致）
  • 配置文件記錄（資料庫連線設定、網路配置）
  • 安全設定確認（防火牆、使用者權限架構）
  • 備份與還原功能已設置

注意：IQ 確認的是「正確安裝了什麼」，不是功能是否正常運作。

LIMS 系統 OQ：運作確認

目的：確認系統的所有配置功能在預期條件下正常運作

測試內容（對應 FRS 中每一個功能需求）：

  • 樣品接收工作流程測試（正常案例 + 邊界案例）
  • 測試方法配置驗證（計算邏輯、規格限制）
  • 電子簽名功能（簽名、意義、不可否認性）
  • 稽核追蹤（記錄創建、修改、刪除）
  • 使用者權限隔離（QC 人員不能修改方法規格）
  • 儀器接口數據傳輸測試

OQ 是最耗時的階段，通常佔驗證工作量的 60-70%。

LIMS 系統 PQ：性能確認

目的：用真實使用場景（end-to-end）確認系統在實際操作條件下符合使用目的

測試方式：

  • 由實際使用者（QC 分析師、樣品接收人員）執行「Day in the Life」測試
  • 使用真實樣品類型和典型批次數量
  • 完整工作流程：樣品登錄 → 指派 → 分析 → 審查 → 報告 → 批次放行
  • 跨部門整合測試（LIMS ↔ ERP 批次記錄系統）

PQ 回答的問題是：「這套系統真的能支撐我們的業務嗎？」

風險為基礎：不是所有系統都需要 PQ

GAMP 5 不強制要求每個系統都必須有 IQ/OQ/PQ 三階段。低風險 Category 3 系統（如辦公室文書工具）可能只需要安裝驗證 + 有限功能測試。測試的深度由風險決定，不由傳統慣例決定。

D5.5 Test Planning/Coverage & Test Cases

D5.5.1 Test Management Processes

Early and efficient testing, during development as well as during the verification phase, improves defect detection and reduces the occurrence of defects surviving into the operational phase.

A test management process may include:

  • Any special prerequisites to testing
  • Identifying what combination of scripted or unscripted, manual or automated testing will be used
  • The personnel required for each test or group of tests
  • The expectations regarding test evidence (including the recording and reporting of test pass/fail results)
  • How the environment in which the testing is to occur will be identified and recorded
  • How the progress and outcome of testing will be monitored
  • How test cases and completed tests will be reviewed and approved

非腳本式測試的最低文件要求：

  • 由誰測試（by whom）
  • 何時測試（when）
  • 發現了哪些問題（any issues found）

探索性測試 = 製程開發中的 DOE

在製程開發中，實驗設計 (DOE) 讓你系統性地探索參數空間，發現意料之外的交互作用。探索性測試類似：測試者用自己的知識系統地探索軟體，發現腳本式測試可能遺漏的邊界案例。

截圖過多不是好事

GAMP 5 明確批評「prolific screenshots」的做法。原則：

  • 通過 (Pass)：記錄「Pass」即可，不需截圖
  • 失敗 (Fail)：記錄「Fail」+ 描述實際結果 vs 預期結果的差異
  • 截圖只在需要「前後對比」的複雜數據場景下才有價值

D5.5.3 Executing Tests

The quality of testing is significantly impacted by the knowledge and skill of the tester, therefore:

  • Day in the life testing needs to be executed by a skilled business process user
  • All other manual testing needs to be executed by a skilled software tester; where unscripted testing is used, knowledge of the system under test is also essential
  • All staff responsible for test execution, including end users, should be trained in test procedures and should be able to demonstrate sufficient confidence in operating the system under test
  • Training should be documented

The identity of the tester should be recorded for all executed test cases. It is disproportionate and unnecessary to use test witnessing or to require the tester to initial every test step.

Automated test tools require a documented assessment of their adequacy prior to use.

Test Evidence — Critical Thinking

It is recommended to apply critical thinking when defining the need for test records and evidence:

  • Ensure that even where testing is not scripted in detail, the objective of the test remains clear, and the results record adequately what testing was carried out
  • Ensure that test evidence is only collected for proving steps that are not inherently covered by evidence from another test
  • Adopt an exception-reporting approach: if the system response matches expected results, a simple "Pass" can be recorded. If the system responds unexpectedly, record both a "Fail" and a description of how the response differed
  • Recognize that routine requirements for screenshots as objective evidence bring little value to verification activities

D5.5.4 Reviewing and Reporting Completed Tests

Completed testing should be reviewed to verify that the computerized system is fit for its intended use. While test evidence should be reviewed to confirm completeness and accuracy, there is little value in:

  • Looking for minor errors in documentation that have no impact on patient safety, product quality, or data integrity
  • Scrutinizing test evidence in extreme detail to assess whether the tester has exactly followed the test instructions for non-critical functionality

Unusual patterns of test failures associated with particular authors and/or individual testers should be investigated to determine whether there are any wider testing implications.

Test reports should be produced that summarize activities and findings and state the final conclusions.

D5.6 Types of Testing

D5.6.1 Test Types Unique to Unscripted Testing

  • Ad-hoc testing — unscripted testing performed without planning or predefined documentation. Aimed at finding defects as early as possible.
  • Error guessing — a testing technique designed to expose anticipated and potential defects based on the specialist tester's knowledge and experience of failure modes.
  • Exploratory testing — experience-based testing where the tester spontaneously designs and executes tests based on existing specialist knowledge, prior exploration, and typical common software behaviors and types of failure and defects.
  • Day in the life testing — allows users of the system to carry out their normal day-to-day activities to uncover issues not identified by scripted testing. These testers need real-life business process experience and knowledge.

D5.6.2 Other Test Types (Scripted or Unscripted)

  • White box testing (structural testing) — test cases identified based on the internal structure (source code) of the component or system.
  • Black box testing (functional testing) — based on an analysis of the specification, without reference to internal structure.
  • Normal Case testing — challenges the system's ability to do what it should do, including triggering alerts and error messages per specifications.
  • Negative Case testing — challenges the system's ability to correctly prevent actions that specifications state should be prohibited or blocked.
  • Repeatability testing — challenges the system's ability to repeatedly do what it should.
  • Performance testing — challenges the system's ability to perform as fast and effectively as required by specifications.
  • Volume/Load testing — challenges the system's ability to manage high loads. Typically automated. Required when system resources are critical.
  • Structural/Path testing — challenges a program's internal structure by exercising detailed program code.
  • Smoke testing — superficial testing to check critical functionality as a measure of system build stability.
  • Regression testing — challenges the system's ability to still do what it should after being modified; verifies that portions not involved in the change were not adversely affected.

D5.6.3 Contractual/Acceptance Testing: FAT and SAT

Acceptance tests are a predefined set of functional tests that demonstrate fitness for intended use. Acceptance may be carried out in two stages:

  • Factory Acceptance Tests (FAT) — performed at the supplier site before delivery to show that the system is working well enough to be installed and tested on-site.
  • Site Acceptance Tests (SAT) — show that the system is working in its operational environment and that it interfaces correctly with other systems and peripherals.

These acceptance tests may be leveraged in support of, but do not replace the need for, process qualification including Process Performance Qualification (PPQ) where relevant.

中文解說：測試類型詳解

迴歸測試 (Regression Testing) — 軟體更新後最關鍵的測試

當 LIMS 供應商發布 v4.1 → v4.2 更新時，需要判斷哪些測試必須重跑：

  • 受影響的功能模組：根據供應商提供的 Release Notes 和變更清單，識別哪些功能被修改
  • 相依功能：修改了樣品接收模組，可能影響樣品狀態追蹤、報告模組
  • 核心 GxP 功能：電子簽名、稽核追蹤、批次放行 — 無論更新內容是否涉及，都建議納入迴歸測試
  • 不需重跑的測試：完全未受影響的功能，有書面風險評估支持

FAT/SAT vs IQ/OQ/PQ — 關係釐清

在自動化設備（如無菌灌裝線控制系統）的驗證中：

  • FAT：在供應商工廠確認系統基本功能（接近 IQ + 部分 OQ）
  • SAT：在現場確認系統在實際環境中運作（接近 OQ + 部分 PQ）
  • FAT/SAT 可以被利用 (leveraged) 作為 IQ/OQ 的證據，減少重複測試
  • 但 FAT/SAT 不能取代 PQ（製程性能確認需要在實際生產條件下進行）

負面案例測試 (Negative Case Testing) 常被忽略

許多驗證方案只測試「系統會做什麼」，卻忘記測試「系統是否正確拒絕不該做的事」。

LIMS 負面測試例子：

  • 在檢驗未完成時，系統是否拒絕批次放行？
  • 當輸入超出規格的數值時，系統是否正確觸發 OOS 警示？
  • 無權限用戶嘗試修改已簽名記錄時，系統是否阻止？

D5.7 Test Environments

Testing may take place in different environments during a project, which may include:

  • Development environment — where prototyping or programming takes place
  • Formal testing environment — dedicated environment for structured validation testing
  • Operational environment — the target production environment

The use of multiple environments provides the opportunity to:

  • Assess changes before they are rolled out to production
  • Provide a target for restore testing of backup data
  • Reduce risk to the live operational environment and data

The DevOps environment used in Agile approaches may be replicated into multiple test environments on demand. The test environments, whether on-premise or hosted by the SaaS provider, need to be functionally equivalent for the testing to be representative and effective. Differences should be assessed and mitigated.

It is essential that a record is kept of the environment in which a test case was run, and the current build and configuration of that environment. This record may reside in an automated tool (e.g., continuous integration orchestration).

There is no benefit to repeating testing already performed in another, but still adequately controlled environment just because it was not the formal test environment.

It is the scope, rigor, and coverage of the testing that ensures the quality of the system as implemented. Any significant differences between the environments used for testing should be assessed to determine the equivalency of test results.

中文解說：測試環境管理

為何需要獨立的測試環境？

在生產環境 (Production) 中直接進行測試是 GxP 的重大風險：

  • 測試數據可能污染真實的 GxP 記錄
  • 測試步驟可能意外修改已驗證的配置
  • 測試失敗可能導致生產中斷
  • 難以維持「已驗證狀態」的完整性

環境等同性 (Functional Equivalency) = 分析方法轉移

如果在實驗室 A 驗證了分析方法，要在實驗室 B 使用，需要做方法轉移確認，證明兩個實驗室的結果具有可比性。

測試環境也一樣：在 TEST 環境中完成的測試，若環境配置與 PROD 有差異，需要評估差異是否影響測試結果的有效性。若不影響（例如：審計追蹤功能的測試結果在兩個環境中相同），則無需在 PROD 中重複測試。

SaaS 環境的特殊考量

對於雲端 SaaS 系統（如某些雲端 LIMS），測試環境由供應商管理。關鍵要求：

  • 確認供應商提供的 sandbox/staging 環境與生產環境在功能上是等同的
  • 記錄每次測試時的環境版本（供應商可能隨時更新）
  • 在合約 (SLA) 中明確測試環境的維護責任

D5.8 Leveraging Previous Testing Including Supplier Testing

D5.8.1 Supplier Testing

The more extensive and effective the testing done by the supplier during their life cycle, the less testing the regulated company may need to do. If the supplier testing is confirmed as sufficient to demonstrate that the system functions correctly against supplier specifications, then the regulated company verification activity may be limited to a subset of requirements testing.

Common test types executed during a supplier development life cycle are:

  • Acceptance Testing for Purchased Hardware and Software — subject to acceptance testing before being used for the system's development
  • Unit/Module Tests — stand-alone tests for software components, ensuring readiness for further integration
  • Integration and Functional Tests — testing of integrated software components, sub-systems, and the complete system

In order to leverage supplier testing, the regulated company should first consider:

  • The extent, coverage and quality of the supplier testing
  • How supplier test records can be made available for appropriate assessment and review
  • How traceability between the regulated company's requirements and the supplier's test cases will be established

D5.8.2 Ongoing Life Cycle Maintenance

New releases of the software by the supplier will result in the need for the regulated company to actively maintain the validated state after upgrading. Critical thinking and risk management should be used to determine the extent of the effort required.

As a minimum, the regulated company will need to verify the updated software remains fit for their intended use. The supplier should provide a clear understanding of the changes being implemented. Regression testing of the configuration and workflows — whether manual or automated — may be implemented to confirm the latest software version still provides a system fit for the regulated company's intended use.

D5.8.3 Testing in Multiple Environments

Testing in environments other than the validation or production environment contributes to finding defects and confirming the system is fit for intended use. Where there is no impact to the particular test types arising from the difference in the environments, the testing should be leveraged no matter what environment it occurred in — there is no benefit to repeating that testing in the formal test or production environments.

中文解說：利用供應商測試

供應商測試的「利用」策略

GAMP 5 的核心效率原則：不重複已完成的良好測試。

能夠利用供應商測試的前提：

  1. 供應商評估已完成（Appendix M2）：確認供應商的品質體系和開發流程符合要求
  2. 供應商測試記錄可以被審查：測試計畫、測試案例、測試結果都可取得
  3. 可追溯性已建立：供應商的測試案例可以對應到受規管公司的 URS/FRS 需求

類比：信任經認可的原料供應商

經過認證的原料供應商提供的物料，不需要每批都做全項檢驗——可以依賴 CoA 並只做鑑別測試。

同樣地，已完成良好審計、測試記錄完整的軟體供應商，其測試結果可以被受規管公司部分採信，減少重複測試。

軟體更新後的迴歸測試決策

當供應商發布 patch 或新版本時，受規管公司的決策流程：

  1. 獲取 Release Notes：理解變更內容
  2. 影響評估：這些變更是否影響我們配置的 GxP 功能？
  3. 決定測試範圍：
    • 若只有安全補丁，無功能變更 → 最小測試（如 smoke test）
    • 若有功能變更 → 受影響功能的完整迴歸測試
    • 若有架構重大更改 → 重新執行全部測試
  4. 文件化決策理由：風險評估記錄

D5.9 Testing Applied to Different Software Categories

D5.9.1.2 Points to Consider for All Systems

This is an aide memoire only and does not replace the need to apply critical thinking and a risk-based approach. Test coverage may include:

  • Power failure testing — prevention against loss of critical data; ease of controlled restart
  • System access and security features
  • Data audit trails for creation, modification, and deletion of GxP records
  • Manual data entry features, input validation
  • Electronic signature features
  • Alarms and error messages
  • Critical calculations
  • Critical transactions
  • Transfer of critical data into other packages or systems
  • Interfaces and data transfers
  • Backup and restore
  • Data archival and retrieval
  • Ability to deal with high-volume loads

D5.9.2 Category 3: Standard Product Testing

Software products used off-the-shelf (not configurable, or limited parameterization). Testing should focus on:

  • Installation testing as described above
  • Requirements testing that demonstrates fitness for intended use (may include testing against requirements depending on risk)
  • For very simple measurement systems, regular calibration may substitute for testing
  • Any further or more rigorous tests as a result of risk and supplier assessments

D5.9.3 Category 4: Configured Functionality Testing

Commercially available software configured for a specific business process. Testing should focus on:

  • Installation testing
  • Configuration testing — verifying that the system has been configured in accordance with the agreed specification
  • Functional testing — challenging functionality supporting the specific business process based on risk and supplier assessments
  • Requirements testing — challenging the configuration and user-defined workflows to ensure business process needs have been met

D5.9.4 Category 5: Custom Application Testing

Custom-developed software. Testing should include Category 4 approaches, plus:

  • Code review for new code required as a result of risk assessments (see Appendix D4)
  • Software module testing — testing software modules defined in the design specification
  • Software integration testing — testing that the modules work when operating together

重要的「全系統通用」測試清單

Section Summary — 章節總結：Appendices D4 & D5

Key Takeaways — English

  • D4 Software Development: GxP software must follow a defined development lifecycle with version control, code review, and release criteria. SBOM is a new best practice for tracking software components and dependencies.
  • Code Review: Source code review has three objectives — verify coding standards, verify design compliance, and identify defects. Review must be performed by an independent person. Static analysis tools can complement manual review.
  • D5 Test Strategy: Risk-based approach is mandatory. Not all functions require the same depth of testing. Critical thinking must be documented.
  • IQ/OQ/PQ in CSV: IQ verifies correct installation and configuration baseline. OQ tests all configured functions against FRS. PQ uses real-world end-to-end scenarios by business users to confirm fitness for intended use.
  • Scripted vs Unscripted: Both are valid. Scripted testing is pre-approved and provides the basis for regression testing. Unscripted testing (exploratory, day-in-the-life) improves defect detection. Unscripted does NOT mean undocumented.
  • Defect Management: Critical defects require fix before go-live. Major defects may allow conditional release with documented workarounds and QA-approved risk assessment. Minor defects can be deferred with documentation.
  • Regression Testing: After software updates, determine test scope by: supplier release notes, impact assessment on GxP functions, and risk-based decision on which tests to re-run.
  • Test Environment: A separate validated test environment is essential. It must be functionally equivalent to production. Record the environment version for every test execution. Testing in other environments (dev, supplier) can be leveraged if equivalency is assessed.

重點總結 — 中文

  • D4 軟體開發：GxP 軟體必須有定義的開發生命週期，涵蓋版本控制、代碼審查、發布標準。SBOM（軟體物料清單）是第二版新增的最佳實踐，用於追蹤軟體元件和相依關係。
  • 代碼審查：三個目標（驗證編碼標準、設計合規性、缺陷識別），必須由獨立人員執行。靜態分析工具可補充人工審查。
  • D5 測試策略：風險為基礎是必要條件。批判性思維的理由必須文件化。並非所有功能都需要相同程度的測試。
  • 電腦化系統的 IQ/OQ/PQ：IQ 確認安裝和配置基線；OQ 對照 FRS 測試所有已配置功能；PQ 由業務使用者用真實場景執行端對端測試，確認符合使用目的。
  • 腳本式 vs 非腳本式測試：腳本式測試需事前批准，是迴歸測試的基礎；探索性/Day-in-the-Life 測試提升缺陷發現率。非腳本式不等於未文件化。
  • 缺陷管理：嚴重缺陷 → 修復再放行；主要缺陷 → 有條件放行（需 QA 批准風險評估 + 暫時工作程序）；次要缺陷 → 記錄後延後修復。
  • 迴歸測試：軟體更新後，依據 Release Notes、GxP 功能影響評估、風險決策，確定需要重新執行哪些測試。
  • 測試環境管理：獨立的已驗證測試環境是必要的，必須與生產環境功能等同。每次測試都需記錄環境版本。已在其他環境完成的測試，在評估等同性後可被利用，無需重複。
