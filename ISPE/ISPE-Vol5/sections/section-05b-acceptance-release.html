7.3.2.2 Factory Acceptance Test (FAT): Installation Tests

p56 — Earliest verification at vendor site; shipping agreements; FAT vs SAT scope

Original Text — English

7.3.2.2 Factory Acceptance Test: Installation Tests

The FAT is inspection/testing of the system at the vendor (system manufacturer) site. The objective of the FAT is to provide testing and documentation at the earliest possible time, to verify that the vendor has fulfilled contractual responsibilities and the system meets the requirements. This allows for quicker and more efficient remedy of any failings and avoids the delays to schedule that would result from discovering problems later, on-site.

After completing the FAT, the owner representative agrees with the vendor whether the system is ready to be shipped or on pre-conditions for shipping. Agreements are documented and approved by the vendor and owner.

Where a system needs to be disassembled for shipping, the process should be documented. Details should include how the system will be disassembled, any marking or labeling of components, and how it will be reassembled and tested on-site. This information provides the rationale for:

  • Testing that can be accepted from the FAT
  • Testing that will be repeated as part of the SAT
中文解說 Commentary

核心概念：FAT（工廠驗收測試）

FAT（Factory Acceptance Test）是在廠商場所進行的安裝/功能測試，目的是「越早發現問題越好」。在設備出廠前發現缺失，修正成本最低、對工期衝擊最小。

FAT 的法律意義：通過 FAT 代表廠商已履行合約責任，雙方共同簽署文件作為憑證。

比喻：FAT = 出廠品管

想像你訂製一台客製化設備，FAT 就是在工廠「出貨前」的驗收。如果你把設備運回公司才發現問題，不但要支付運費來回，還要等修復時間——這在製藥廠的專案時程裡可能代價極高。

關鍵：拆卸運輸決定 FAT vs SAT 的測試分工

若設備需要拆卸後運輸（如大型 WFI 系統），則在廠商已測試過的項目中，部分需要在 SAT 重新測試（因為重新組裝可能影響原有的校驗結果）。相反，不受拆卸影響的測試結果可沿用 FAT 數據。

這個拆卸/重組計畫必須在 FAT 時就事先文件化，作為 SAT 測試範圍的依據。

CDMO 實務應用

CDMO 接受客戶委託時，若客戶要求見證 FAT（通常是 Phase 3 或商業製造設備），必須事先確認：觀察人員資格、NDA 保護、拍照/文件限制，以及 FAT punch list 的所有權歸屬。

7.3.2.3 Receipt Inspection & 7.3.2.4 Site Acceptance Test

p56–p57 — Delivery documentation; tagging; SAT at owner site; turnover conditions

Original Text — English

7.3.2.3 Receipt Inspection

Receipt Inspection (RI) provides documentation that delivered components, materials, or systems conform to the requirements, e.g., manufacturer, model, serial number as applicable with the required supporting documentation. This is typically done immediately upon delivery of the component to the owner site.

RI also provides documentation that all components have been received in good condition and are tagged correctly, where applicable.

Any items that do not pass RI are separated for issue resolution. Items that pass RI are stored under suitable conditions for installation.

7.3.2.4 Site Acceptance Test: Installation Tests

The SAT is the testing of the system at the owner organization site. The objective of the SAT is to provide testing and documentation of the installed system, to verify that the vendor has fulfilled contractual responsibilities and that the system meets the requirements prior to contractually accepting the system.

The SAT test plan includes a review of the FAT punch list (items carried over from the FAT that did not meet the acceptance criteria/expected results) and the plan to resolve open items. The SAT also includes reconfirming FAT test results where required due to system disassembly and reassembly.

After completing SAT testing, the owner representative agrees with the vendor whether the system is ready to be turned over to the owner or on pre-conditions for system turnover. Agreements are documented and approved by the vendor and owner. Acceptance of a system with open discrepancies can be made with the approval of the Testing Coordinator and System Owner.

中文解說 Commentary

RI（到貨驗收）的重點

到貨驗收 (Receipt Inspection, RI) 是設備/材料抵達現場後的第一道查核，核對的是「對不對、完不完整、有沒有損壞」。

  • 確認廠牌、型號、序號符合採購規格
  • 確認隨附文件（材質證明、校驗報告等）齊全
  • 確認到貨狀況良好、標籤正確
  • 不合格品需隔離管制，不得進入安裝流程

比喻：RI = 網購收貨拆箱

就像網購收到包裹要當場檢查，RI 必須在到貨時立即執行。等到設備裝到系統裡才發現是錯誤規格，要拆卸更換的代價就大得多了。

含未解偏差的系統仍可驗收

SAT 完成後，若有未關閉的偏差事項，在 Testing Coordinator 與 System Owner 共同批准的前提下，仍可先行驗收並移交。這是實務上常見的做法，但需要明確記錄未解項目及後續解決計畫。

7.3.2.5 Installation Verification & 7.3.3 System Start-Up

p57 — CDE-specific installation verification; 4-step start-up process; vendor training

Original Text — English

7.3.2.5 Installation Verification

Installation verification includes verifying quality requirements related to installation (CDEs as defined through the System Risk Assessment). Example activities include:

  • Verification that the product, ingredient and/or cleaning material contacting parts meet the specifications (materials of construction and finishes, as applicable)
  • Installation of critical instruments
  • Calibration of critical instruments (calibration could also be verified at the start of the functional testing, prior to the commencement of testing)

7.3.3 System Start-Up

System start-up is a GEP approach to ensure that the system is energized and started up in a safe manner. This is also where adjustments to the system, settings to regulate the system, or any balancing needed are made. A prerequisite to system start-up is that the system is mechanically complete without open critical punch list items for safety. System start-up may be conducted both at the vendor facility and after installed on-site.

The vendor operation or start-up manual should be reviewed prior to energizing the system. For systems that include vendor-provided control systems, training should be completed prior to energizing the system. Training of system operators, engineers, and maintenance personnel should be conducted in association with system vendors and properly documented.

System start-up is a four-step process, consisting of:

  1. Verification that the system is complete with all valve switches in the correct position. For stick-built systems and modular systems, ensure that the integrated system (including automation) has been reviewed before it is energized.
  2. Pre-start-up safety review.
  3. Supervised energizing of the system utilities.
  4. Pre-functional check of the system, with focus on safety functions.

Note that Steps 1 and 2 above need to be completed before Step 3. Step 3 needs to be completed prior to Step 4.

Any issues identified during system start-up activities should be verified to not impact future testing and addressed during the following testing steps.

中文解說 Commentary

安裝驗證 (Installation Verification) 聚焦 CDE

Installation Verification 不是全面性的安裝查核——它特別針對系統風險評估中識別出的 CDEs（關鍵設計元素）。

最重要的兩類查核：

  • 材質符合性：與產品/清潔劑接觸的部件，材質是否符合規格（例如：316L 不鏽鋼、電解拋光等）
  • 關鍵儀器校驗：溫度計、壓力計、流量計等的 calibration 是否完成且有效

供應商培訓是通電的前置條件

對於含廠商提供控制系統的設備（如 PLC、SCADA），操作員、工程師、維護人員的培訓必須在通電之前完成，且有文件記錄。這不僅是安全要求，更是 GEP 的基本原則。

CDMO 應用：PSSR（啟動前安全審查）

製藥廠普遍使用 PSSR (Pre-Start-Up Safety Review) 作為正式清單。建議將 ISPE 的四步驟整合進 PSSR SOP，並要求：(1) 安全相關 punch list 全部關閉；(2) 消防/HSE 部門會簽；(3) 培訓記錄附件。

系統啟動四步驟（順序不可顛倒）

ISPE 明確規定啟動步驟必須依序執行，有強制前後依存關係：

步驟內容前提
Step 1確認系統機械完整性與閥門位置無安全相關重大 punch list
Step 2啟動前安全審查 (Pre-PSSR)完成 Step 1
Step 3監督下通電系統公用設施完成 Step 1+2
Step 4功能前查核（聚焦安全功能）完成 Step 3

7.3.4 Operational / Functional Testing

p57–p59 — Full-range testing; FAT/SAT operational tests; PLC/SCADA; qualification OQ

Original Text — English

7.3.4 Operational/Functional Testing

Operational/functional testing verifies that the system functionality matches what has been approved in the specifications (including those identified as CAs) and is suitably documented in a test plan/record for the full desired range of system operations in the target environment.

The sequence of functionalities being tested should follow a planned approach. Functional tests should be executed after completion of all installation tests that can impact the functional tests. Testing begins after system start-up and calibrations defined in the respective plan have been completed.

Testing may be conducted within a single report/test plan or separated into FAT, SAT, commissioning, and qualification phases. Installation and operational testing may be combined in a single document.

The scope of functional testing includes verifying system functionality including mechanical, electrical, automation, and HSE related aspects. Documents, drawings, and operational procedures may be drafted and redlined, and discrepancy lists may be established.

Typically, the final step of functional testing is a full operational testing — a full load test of the system to encompass worst case conditions. This test may need to be deferred if it is not possible to provide the relevant conditions. Final tuning of control loops may need to be delayed until system balancing has been performed (e.g., air or water balancing) or until enough system loads are available.

Functional testing activities include, for example:

  • Testing integrated automation controls and equipment for the whole system — both normal and alarm/failure modes
  • Testing analog control loops at a range of values representing the intended system use — encompassing the anticipated range of normal operation, including upper and lower operating limits (worst case conditions) without inducing process failure
  • Confirming the system can handle predictable simultaneous use scenarios for utility systems
  • Determining and recording manual component settings (e.g., valve positions, limit switch/screw positions, control switch settings)

7.3.4.1 Factory Acceptance Test and Site Acceptance Test: Operational Tests

After installation testing and system start-up, the FAT and SAT test documents may include execution of operational testing.

7.3.4.2 Qualification Operational Tests

Operational tests that demonstrate the ability of the system to meet the CAs/CDE functionalities are considered part of the qualification of the system and require Quality Unit oversight. Complex or novel systems may require formal system integrated operational testing to document PQ/PPQ operational criteria/control strategy.

中文解說 Commentary

功能測試的核心原則

功能測試的目標是驗證系統在整個預期操作範圍內——包括正常條件與最差情境（worst case）——皆符合規格。

關鍵要素：

  • 必須在安裝測試與校驗完成後才開始
  • 涵蓋機械、電氣、自動化（PLC/SCADA）、HSE 等面向
  • 最終步驟通常是全負載測試（Full Load Test）

比喻：worst case 測試 = 壓力測試

就像汽車要進行極端天氣路試，功能測試必須覆蓋上下限操作條件——例如 HVAC 在夏天最熱、冬天最冷時是否都能達到設計換氣次數。只在「平均值」條件下測試是不夠的。

7.3.4.2 QOT：需要 QA 介入的關鍵區分

Qualification Operational Tests (QOT) 與一般功能測試的最大差異：

  • 針對 CAs/CDEs 的功能測試 = 屬於確效活動
  • 必須有 Quality Unit（QA）監督
  • 測試結果的偏差需走 QA 核准流程

換言之，同一台設備的功能測試，與 CA 相關的部分屬於 OQ（操作確效），QA 必須介入；與 CA 無關的部分只是 commissioning，工程師可自行完成。

思考練習

製藥廠有一條蒸汽管路系統。蒸汽壓力是 CDE（影響滅菌效果）。問：測試「蒸汽管路自動切換閥門在異常壓力時的連鎖動作」屬於 QOT 還是一般功能測試？該如何處理？

答：蒸汽壓力是 CDE，與之相關的閥門連鎖動作直接影響滅菌安全性，因此屬於 QOT，需要 QA 監督，偏差需走確效偏差流程處理。

7.3.5 Performance Qualification Testing

p59 — PQ with product/placebo; operational SOPs required; systems needing PQ

Original Text — English

7.3.5 Performance Qualification Testing

Performance Qualification (PQ) testing is the documented verification that manufacturing systems perform effectively and reproducibly, based on the approved process or analytical method and product specification, applying either original product or representative placebo. Cycle development and/or engineering tests are often necessary prior to performing PQ testing.

Completion of installation and operational testing (as applicable) to ensure the systems are operating correctly is required before performing cycle development; PQ testing can then be conducted. Operational SOPs should be used during PQ.

Examples of systems where PQ is usually required before acceptance and release include:

  • Autoclaves
  • Filling machines
  • Pharmaceutical water and steam systems
中文解說 Commentary

PQ（性能確認）的三大特徵

PQ（Performance Qualification）是確效三步驟中最接近「真實使用」的最後一關：

  1. 使用真實產品或替代品（placebo）——不再只是用水或模擬介質
  2. 驗證重現性——通常需要多次重複（如滅菌確效三次）
  3. 在操作 SOP 下執行——驗證「人員按照 SOP 操作」也能達到規格

PQ 的先決條件鏈

ISPE 規定的順序非常嚴格：

IQ 完成 → OQ 完成 → 工程測試/週期開發 → 制定操作 SOP → 執行 PQ

若跳過中間步驟（如未先完成週期開發就直接執行 PQ），稽核時將面臨重大問題。

比喻：PQ = 考試的「上機實作」

IQ 是核對「工具有沒有裝好」，OQ 是「操作手冊說得到嗎」，而 PQ 是「真正考試時，考生按照 SOP 操作，結果符合規格嗎」。PQ 失敗了，表示真實操作條件下系統仍達不到要求，整個放行流程必須暫停。

常見需要 PQ 的系統

系統PQ 重點
高壓滅菌鍋 (Autoclave)溫度分佈、F0 值重現性（≥3次）
無菌充填機無菌工藝模擬 (Media Fill/APS)
製藥用水系統水質連續監測、菌落數、TOC

7.4 Discrepancy Management

This section describes how to handle C&Q test results that do not meet the expected results or acceptance criteria, also referred to in this Guide as discrepancies. Discrepancies related to CAs/CDEs are critical and require Quality Unit approval prior to closure.

Expected results are engineering expectations; failure to meet these should be reported to the design authority/SME who can accept them with a suitable justification or initiate engineering changes. The design authority has the legal obligation to ensure the system meets codes.

Acceptance criteria are those testing/inspection results that must be met for a system to meet the requirements, which includes quality requirements. Examples:

  • Loop return pressure in a WFI system (negative pressure = risk of contamination if a valve is opened)

Type 4 vs Type 5 的關鍵差異

Chapter 8: Acceptance and Release — Introduction (8.1)

p61 — Formal review of all testing; initiation of qualified state; QA-managed change control

Original Text — English

8 Acceptance and Release

8.1 Introduction

This chapter provides guidance on the acceptance and release phase which is where all testing is formally reviewed and accepted. The conclusion of the acceptance and release phase is that the system can be released for:

  • Manufacturing operations
  • The next stage of qualification (e.g., for a WFI system)
  • Use in the manufacture of PPQ or commercial batches

Acceptance and release represents the initiation of the "qualified state" and the beginning of QA-managed Quality Change Control (see Chapter 12), continuous performance monitoring, and periodic review. The project team transitions system ownership to operations/engineering.

The acceptance and release phase should occur as a cooperative review between the appropriate SMEs, including a representative from the department that will accept care, custody, and control of the system. The SMEs should review the results from an overall technical perspective.

If the system includes CAs/CDEs (direct impact), then the Quality Unit needs to be part of the acceptance and release review and approval, focusing on CAs/CDEs. Acceptance of a system with open discrepancies should be a Quality Unit decision. Typically, all discrepancies have to be addressed; however, non-product impacting discrepancies may remain open with an appropriate approved action plan.

The acceptance and release phase is not intended to be a catchall for closeout of all system or project documentation.

中文解說 Commentary

「確效狀態」啟動的法規意義

QA 簽署驗收與放行文件的那一刻，正式宣告系統進入確效狀態（Qualified State）。這個時間點在法規上具有以下意義：

  • 系統的任何後續更改必須走 QA 管理的品質變更控制
  • 啟動持續績效監測程序
  • 啟動定期審查週期（通常每年或每3年）
  • 系統所有權正式由專案團隊移交給操作/工程部門

比喻：確效狀態 = 房屋完工驗收後的交屋

在驗收放行前，設備是「建造中」——工程師可以自由修改。一旦 QA 簽字放行，就像房屋交屋後，任何改動都需要「裝修許可」（變更控制）。沒有經過正式的 QA 核准變更控制就動到系統，等同於未申報裝修——稽核時這是重大缺失。

「非影響產品品質」偏差可帶著放行

不是所有偏差都必須在放行前關閉。若偏差不影響產品品質，可以在以下前提下暫時開放：

  • 已有書面記錄的解決行動計畫
  • 指定負責人與完成期限
  • QA 正式批准此例外

8.3 Requirements to Complete the Acceptance and Release Phase

p62–p64 — Testing docs; discrepancy closure; engineering change closure; quality system elements

Original Text — English

8.3 Requirements to Complete the Acceptance and Release Phase

In order to complete the acceptance and release phase, the following items are required to be addressed: completion of test documentation, closure of discrepancies and changes, and implementation of quality system elements.

8.3.1 Completion of Testing Documentation

All of the planned testing needs to be successfully executed and the documentation completed. At least one SME review should be documented within the executed tests.

The documentation for completion of testing should be approved by the team of SMEs and include the following information at a minimum:

  • Testing description (component verification, calibration verification, etc.)
  • Testing expected results and if met or not met
  • Testing acceptance criteria and if met or not met
  • Reference to test protocol and execution data (e.g., vendor documents, commissioning)
  • Summary and conclusion

8.3.2 Closure of Discrepancies

Discrepancies should be categorized by potential impact.

  • Discrepancies that do not impact product quality may remain open. The report should contain an indication of any open discrepancies along with potential manufacturing planning implications. These discrepancies should be assigned to a responsible individual with defined responsibility for resolution within a specific timeline.
  • Discrepancies that could impact product quality need to be closed.
  • All discrepancies should be listed with status (closed or open) within the summary report.

8.3.3 Closure of Engineering Changes

All changes should be handled in accordance with change management (as described in Chapter 12). The system changes that have occurred during the testing phase should be closed as part of the acceptance and release phase, unless there is agreement with the Quality Unit for these open changes to be carried forward. All engineering changes should be listed with status (closed or open) within the summary report.

8.3.4 Implementation of Quality System Elements

For systems that contain CAs/CDEs, the implementation of quality systems is required prior to releasing a system for manufacturing operations. These aspects are referred to as quality system elements and may include (as applicable), but are not limited to:

  • Availability of SOPs for operation of the system
  • Calibration/maintenance system and associated procedures
  • Mechanism for tracking system use and maintenance (e.g., logbook)
  • Vendor documentation (e.g., maintenance requirements, spare parts)
  • Operator training within the training quality system
  • Completion of any other organization requirements to add the system into the operating facility

A statement that all required quality system elements are in place should be included in the summary report for systems with CAs/CDEs. If a system is released with outstanding items (requires Quality Unit approval), a quality system should be in place to track these items to closure with an appropriate level of Quality Unit oversight.

8.4 Summary Report Organization — 4 Example Options

p63–p66 — Commissioning CSR; Qualification QSR; Change Control; Large Projects A&R Report

Original Text — English

8.4 Summary Report Organization

The acceptance and release process is completed through a summary report. The contents and complexity of the report may vary, but it needs to include SME review and approval of: closure of testing documentation, discrepancies, engineering changes, implementation of quality system elements (as required), and documented formal release of the system.

The summary report can be accomplished in several different ways depending on the scale and complexity of the project. A separate document for the summary report is not always necessary.

8.4.1 Example Option #1: Commissioning Summary Report – Systems with No CAs/CDEs

For systems with no CAs/CDEs (not direct impact), the acceptance and release can be included within the Commissioning Summary Report, which includes:

  • Completion of testing documentation
  • Closure of punch list items and discrepancies
  • Closure of engineering changes
  • Confirmation of documentation in place to ensure the system can operate and be maintained properly
  • Clear statement of formal release of the system

8.4.2 Example Option #2: Qualification Summary Report – Systems with CAs/CDEs

For systems with CAs/CDEs, where all of the requirements are ready at about the same time, the acceptance and release can be included within a single C&Q Summary Report, which includes:

  • Completion of testing documentation
  • Closure of discrepancies
  • Closure of engineering changes
  • Quality System Elements implementation (or reference to change control record)
  • Clear statement of formal release of the system (or reference to change control record)

8.4.3 Example Option #3: Formal Release with Use of Change Control – Existing Facility

When new systems or changes to existing systems are implemented through QA-managed Quality Change Control, some or all of the requirements may be contained within the change control. For existing facilities, it is common to verify the implementation of quality system elements and document the formal release of the system in the change control rather than in the Qualification Summary Report. The use of the change control allows for closure of the testing phase in a timely manner.

8.4.4 Example Option #4: Acceptance and Release Report – Large Projects with Interconnected Systems

For large projects with interconnected systems, multiple reports may be used to complete the acceptance and release phase. An overall acceptance and release report would reference the completed summary reports for each of the interconnected systems, document completion of all quality system elements, and include the formal release statement.

One strategy may be to release per processing area (e.g., cell culture) after all testing is completed and closed out with separate summary reports.

Note: The grouping strategy may complicate future changes. The grouping would also require presenting information from all systems during regulatory audits even when only a single system is requested. The advantages are that qualification would more closely reflect the grouping created by the process risk assessment as well as the batch record structure and eliminate potential gaps with the integration of systems.

8.5 Acceptance and Release Completion & 8.6 Traceability Matrix

p67–p68 — QA signs = qualified state begins; Traceability Matrix as optional engineering tool

Original Text — English

8.5 Acceptance and Release Completion

Approval and release is completed when the Quality Unit approves and signs the summary report, which could be the Qualification Summary Report, change control closure, or Acceptance and Release Summary Report. This final approval releases the system for use. Refer to Section 6.4 for an example document approval matrix.

Quality Unit acceptance and release represents the initiation of the "qualified state" and the beginning of QA-managed Quality Change Control (described in Chapter 12), continuous performance monitoring, and periodic review.

8.6 Traceability Matrix

A Traceability Matrix may be used as a tool to identify CAs/CDEs, acceptance criteria, and includes reference to the testing document (planned or completed). Best practice is to create the matrix early in the project or as soon as the CAs/CDEs are identified. The Traceability Matrix can be used throughout the project as an engineering tool to manage DR, testing activities, results, and documentation that ensures all CAs/CDEs have been met. It provides traceability for each URS requirement by design and through testing.

As such, the Traceability Matrix is not required to be approved. Approvals are completed for the test protocols and the Qualification Summary Report or Acceptance and Release Report. The Traceability Matrix is not a required standalone deliverable.

For smaller projects, a Traceability Matrix may not be needed. It is considered a useful project management tool when testing is planned to be completed in several different documents or stages conducted by different companies. It is recommended to keep the amount of information managed in this document to a minimum to avoid an unmanageable document size when printed and to simplify tracking of changes.
