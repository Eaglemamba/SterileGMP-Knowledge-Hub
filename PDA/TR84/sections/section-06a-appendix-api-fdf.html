學習目標 Learning Objectives

  1. 理解如何依據資料關鍵性（高／中／低）與現有管控層級，將製程資料定位於 9-Box 脆弱性矩陣的正確位置。

Example 1: High Criticality — API Reaction Controls Impurity Level

高關鍵性：原料藥反應器操作控制雜質生成 | Table 8.1-1🔍 | p42

Original Text (English)

Table 8.1-1🔍 provides worked examples for high-criticality data in API manufacturing. The reactor operation controls impurity levels, directly impacting a Critical Quality Attribute (CQA) through a Critical Process Parameter (CPP). Three sub-examples (1a, 1b, 1c) illustrate progressively improved data integrity controls, moving from a manually controlled reactor with paper records, to a PLC-controlled reactor with paper records, to a PLC-controlled reactor with a fully electronic batch record (EBR).

中文解析

核心情境

反應器操作直接控制雜質生成，屬於 CPP（關鍵製程參數, Critical Process Parameter），影響 CQA（關鍵品質屬性, Critical Quality Attribute）。這是 API 製造中最高風險的資料類型，位於 9-Box 矩陣頂列。

三個子案例展示同一操作在不同數位化程度下的資料脆弱性結果，是理解「技術升級如何降低風險」的最佳範本。

Table 8.1-1🔍: Example 1 — High Criticality API Reaction
中文解析

DI 問題核心：手動抄錄的斷鏈風險

1a 問題：操作員從儀表板讀取數值後手寫到紙本批次記錄（BPR, Batch Production Record）。這個「人工轉錄」步驟是資料完整性的最大漏洞——數值可能被錯誤抄寫、事後補填、甚至刻意竄改，而無法追溯原始來源。

1b 改善：加入 PLC（可程式邏輯控制器）後，設備本身已有電子記錄，但數據仍需人工從 HMI（人機介面）抄錄到紙本 BPR。問題縮小但未消除，且 PLC 本身若無稽核軌跡設定，電子資料仍可能被覆寫。

1c 最佳：EBR（電子批次記錄, Electronic Batch Record）直接與 PLC 整合，數據自動寫入，人工介入降至殘餘水準。稽核軌跡完整，任何修改均有記錄。

比喻：溫度計讀數的三種記錄方式

想像病房護理師每小時量體溫：

  • 1a：看著水銀溫度計，憑記憶一小時後補寫在紙上。
  • 1b：有電子體溫計顯示數值，但護理師仍需手抄到護理記錄。
  • 1c：電子體溫計直接藍牙傳輸到電子病歷，護理師確認即可。

製藥界要求的就是 1c 的可靠程度——尤其當這個「體溫」直接關係到病人用藥安全（= 產品 CQA）時。

稽核警示：1b 的常見陷阱

許多 CDMO 處於 1b 狀態：有 PLC 但沒有設定稽核軌跡，或稽核軌跡功能被關閉「以節省儲存空間」。FDA 483 觀察項目中，這是高頻缺失之一。關鍵提醒：PLC 有記錄能力不等於 DI 合規——必須啟用、驗證、並確保記錄不可被刪除。

9-Box 定位：Example 1

資料關鍵性（縱軸）× 管控水準（橫軸）
─────────────────────────────────────
1a: High × Low   → Significant（紅）
1b: High × Medium → Moderate（橙）
1c: High × High  → Acceptable（綠）

注意：高關鍵性資料在任何管控水準下，
最高只能達到 Acceptable（無法達到 Negligible）

思考問題

  1. 你的 CDMO 現有 API 反應器屬於 1a、1b 還是 1c？若是 1b，PLC 的稽核軌跡是否已驗證並啟用？
  2. 若客戶要求你提供反應溫度的原始電子記錄（raw data），你能提供嗎？從哪個系統匯出？

CDMO 應用：如何從 1b 升級到 1c

  • 盤點現有 PLC 系統是否支援 21 CFR Part 11 或 Annex 11 稽核軌跡功能
  • 若支援但未啟用，立即安排 CSV（電腦化系統驗證）變更評估
  • 評估 EBR 系統（如 Syncade、MasterControl、Tulip）的整合可行性與投資報酬率
  • 短期過渡方案：為 HMI 列印輸出設計受控的「黏貼驗證」程序，確保列印品即時、完整、不可竄改

Example 2: Medium Criticality — Drying of API (LOD not a CQA)

中關鍵性：原料藥烘乾操作（乾燥失重非 CQA）| Table 8.1-2🔍 | p42–43

Original Text (English)

Example 2 covers the oven drying of an API (Table 8.1-2🔍). The loss on drying (LOD) of the API is not a CQA and the drying operation is not a CPP, but the LOD does impact the flow characteristics of the API, which affects the downstream processing of the API into the finished product. Therefore, the data regarding the drying of the API is of Medium criticality and belongs in the middle row of the 9-Box vulnerability grid.

Example 2a describes Medium-criticality data with the manual operation of the drying oven, manual data capture, and manual transcription of data into a paper manufacturing record. In addition, the existing data integrity controls are Low-level, with reliance on second-person witnessing of the manufacturing operation, human oversight of the manufacturing process, and human review at release. The Medium criticality of the data plus the Low level of controls yields Moderate data vulnerability (Orange), as described in Section 4.3.

Example 2b considers the same Medium-criticality data. Here, though, the data integrity controls have increased with the addition of a validated PLC, and the inclusion of the PLC printouts with the BPR for the review and release process. The additional controls yield an Acceptable level of data vulnerability (Green).

Example 2c illustrates the same Medium-criticality data but with more than the required controls in place: a validated EBR with automatic data capture, the inability to erase any data, activated audit trails, automated data archiving, and the ability to trend the electronically stored production data. These controls yield a Negligible level of data vulnerability (Blue).

中文解析

中關鍵性的定義邏輯

LOD（乾燥失重, Loss on Drying）不直接是 CQA，但影響 API 的流動性，進而影響後續製劑製程（混合、壓片）。這種「間接影響品質」的資料屬於中關鍵性，位於 9-Box 矩陣中列。

關鍵洞察：中關鍵性資料在低管控時仍會達到 Moderate（橙色）——代表即使不是 CQA，忽視 DI 管控仍有顯著風險。

2c 的重要啟示：超過要求的控制

2c 的控制水準（EBR + 稽核軌跡 + 自動存檔）超過中關鍵性資料的最低要求。結果達到 Negligible（藍色）。TR84 特別指出：這並非「必要」，而是展示技術能力上限。對於 CDMO，這意味著若已有整合式 EBR 系統，中低關鍵性資料可「順帶」達到最高合規水準，無需額外成本。

中文解析

比喻：廚房麵粉的濕度管理

LOD 對 API 流動性的影響，就像麵粉含水量對揉麵的影響：麵粉本身的品質（CQA）沒問題，但太潮濕的麵粉在後段製程會結塊、難以定量——影響的是製程一致性，而非最終成品的直接安全性。這就是「中關鍵性」的核心含義。

資料關鍵性（縱軸）× 管控水準（橫軸）
─────────────────────────────────────
2a: Medium × Low    → Moderate（橙）
2b: Medium × Medium → Acceptable（綠）
2c: Medium × High   → Negligible（藍）

中關鍵性資料在高管控下可達 Negligible，
優於高關鍵性（最高只到 Acceptable）

LOD 管控的 DI 重點

  • 溫度記錄：烘乾溫度與時間是主要記錄對象，需確保儀器校驗狀態有效
  • 取樣測試記錄：LOD 測試結果本身也需完整稽核軌跡，特別是重測（retest）情形
  • 批次間一致性：趨勢分析（trending）在 2c 的 EBR 環境中自動可行，是預防性品質管理的基礎

思考問題

  1. 你的 LOD 測試儀（如水分分析儀）是否有電子記錄功能？測試結果是直接匯入系統還是手抄？
  2. 若客戶審計要求提供某批次 API 烘乾的完整製程數據趨勢圖，你能在 24 小時內提供嗎？

CDMO 應用：中關鍵性資料的合理投資策略

對於中關鍵性資料，2b（PLC + 列印輸出）通常是最具成本效益的合規解決方案。升級到 2c 的 EBR 整合是好的，但應以系統整合的整體效益為決策依據，而非單純為了提升 DI 等級。

Example 3: Low Criticality — Small Molecule API Batch-to-Batch Cleaning

低關鍵性：小分子原料藥批次間設備清潔 | Table 8.1-3🔍 | p44–45

Original Text (English)

Example 3 depicts the batch-to-batch cleaning of API manufacturing equipment of a small molecule during a product campaign within the validated campaign length (Table 8.1-3🔍). Since this type of data is neither directly nor indirectly related to a CQA or CPP, it is considered routine GMP data and of Low criticality, according to the definitions in Section 4.2.1. Therefore, this data resides in the bottom row of the 9-Box vulnerability grid.

In Example 3a, the cleaning is performed manually, then the Low-critical data is captured by the operator who transcribes it into the paper manufacturing record. The reliance on second-person witnessing, human oversight of the manufacturing process, and human review at release provide a Low level of data integrity controls. The Low criticality of the data plus the Low level of controls yields Acceptable data vulnerability (Green).

Example 3b illustrates the same Low-critical data but the level of controls in place are more than required. The process is better controlled through a validated clean-in-place (CIP) process. The data integrity controls have increased with the PLC printouts from the CIP system, which are included with the BPR for the review and release process. The additional controls yield Negligible data vulnerability (Blue).

Example 3c illustrates the same Low-critical data but with the addition of a validated EBR with automatic data capture, inability to erase any data, activated audit trails, automated data archiving, and ability to trend the electronically stored production data. The vulnerability of the data remains unchanged, and the vulnerability remains at a Negligible level (Blue). In this example, there are significantly more controls in place than are required.

中文解析

低關鍵性的定義重點：「驗證活動期間內」

此案例的低關鍵性前提是：清潔操作在已驗證的活動期間（campaign length）內執行。這個限制條件至關重要——若超出驗證的連續批次數，殘留物的累積可能影響 API 品質，資料關鍵性就會提升。

這提醒我們：DI 風險評估不能只看操作類型，還必須考慮操作的邊界條件是否仍在驗證範圍內。

3a 的重要洞察：低 × 低 = 可接受（非問題）

TR84 明確指出：低關鍵性資料搭配低管控水準，結果是 Acceptable（綠色），不是問題。這挑戰了「所有 GMP 記錄都需要電子化」的誤解。監管者關注的是風險相稱性——過度控制低風險資料是浪費資源，甚至可能引入新的系統複雜性風險。

Table 8.1-3🔍: Example 3 — Low Criticality API Batch-to-Batch Cleaning
中文解析

比喻：餐廳廚具的班次間清洗

同一間餐廳同一天烹調同一道菜，廚具班次間的沖洗記錄屬於「例行清潔」，不需要和跨菜品或跨天的深層清潔（相當於「活動結束清潔」）一樣嚴格的記錄體系。TR84 的低關鍵性邏輯正是如此：在已驗證的使用週期內，簡單記錄已足夠。

3b 與 3c 的差異：Negligible 已是頂點

3b（CIP + PLC 列印）和 3c（EBR 整合）都達到 Negligible（藍色）。這說明對低關鍵性資料，從 3b 升級到 3c 在 DI 脆弱性上不帶來額外效益——只有在系統整合或效率提升上才有意義。TR84 的訊息是：不要為了升色塊而過度投資。

9-Box 定位：Example 3

資料關鍵性（縱軸）× 管控水準（橫軸）
─────────────────────────────────────
3a: Low × Low    → Acceptable（綠）✓ 已合規
3b: Low × Medium → Negligible（藍）超過要求
3c: Low × High   → Negligible（藍）超過要求

低關鍵性 + 低管控 = 已合規。
紙本 BPR 對此類資料是完全可接受的解決方案。

思考問題

  1. 你的 CIP 系統是否已驗證？驗證報告中是否明確定義了活動期間（campaign length）的上限？
  2. 若你的設備清潔記錄目前是紙本，是否有必要升級為電子系統？依據 TR84，答案應如何論證？

CDMO 應用：清潔記錄的風險分類管理

  • 建立清潔記錄的風險分類清單：區分「活動期間內批次間清潔」vs「產品切換清潔」vs「定期深層清潔」
  • 對批次間清潔維持紙本記錄是合理的；對產品切換清潔需考量是否有 CQA 影響（可能升為中關鍵性）
  • 確保每份清潔記錄都有「清潔類型」和「活動批次序號」，以利追溯是否仍在驗證範圍內

FDF Examples — Finished Dosage Form

最終劑型 (Finished Dosage Form) 案例 | Examples 4–6 | p46–49

FDF 案例概述

第 8.2 節使用相同製程（篩分操作中的濕度監測）作為跨三組案例的統一背景，藉此清晰呈現「資料關鍵性改變」（而非操作改變）對 9-Box 定位的影響。Examples 4、5、6 分別對應高、中、低關鍵性，每組同樣含三個控制水準子案例（a/b/c）。

藥物的吸濕性（hygroscopicity）決定了濕度是否為 CPP，進而決定資料關鍵性——這是 FDF 案例組的核心設計邏輯。

Example 4: High Criticality — Humidity Monitoring During Sieving

高關鍵性：篩分操作濕度監測（吸濕性藥物）| Table 8.2-1🔍 | p46–47

Original Text (English)

Examples 4–6 demonstrate where High-, Medium-, and Low-criticality data from a sieving operation are placed in the 9-Box vulnerability grid, based on the level of controls in place. The examples are for humidity monitoring during sieving of a drug substance, prior to blending and compression.

In Example 4, humidity is a CPP that will impact CQA (Table 8.2-1🔍). The drug substance is hygroscopic and will degrade with elevated levels of moisture. Therefore, the High-criticality data goes in the top row of the 9-Box vulnerability grid.

Example 4a illustrates High-criticality data with the operator manually inserting analog humidity data from a chart recorder and transcribing this data to a paper manufacturing record. A Low level of data controls exist, relying on a second person to verify the data. The High criticality of the data combined with the Low level of controls creates Significant data vulnerability (Red), as defined in Section 4.3.

Example 4b illustrates the same High-criticality data. In this example, the data controls have increased compared to Example 4a. The humidity data is now digitized and captured using a validated building management system (BMS), using audit trail and access controls. Also, a printout of the humidity data is attached to the manufacturing record. The additional controls yield Moderate data vulnerability (Orange).

Example 4c illustrates the same High-criticality data with the additional controls of a validated electronic manufacturing record with automated data capture, audit trail, and access controls. These additional controls yield Acceptable vulnerability (Green).

中文解析

吸濕性 = CPP 的判定邏輯

藥物具有吸濕性（hygroscopic）時，製程環境中的相對濕度直接影響降解速率，因此濕度成為 CPP，其監測資料屬於高關鍵性。這與同樣的濕度監測在非吸濕性藥物（Example 6）中只屬低關鍵性形成鮮明對比。

設計重點：TR84 刻意用「同一操作、不同藥物屬性」的設計，強調關鍵性評估必須以產品特性為基礎，而非操作類型。

4a 的 Red 警示：最需立即行動

Significant（紅色）是 9-Box 中最高風險等級，代表需要立即採取 CAPA（矯正與預防措施）。若你的 CDMO 正在處理吸濕性藥物，且濕度監測仍依賴紙本圖表記錄，這個案例就是你現在的處境——需要優先升級。

中文解析

比喻：博物館文物的濕度管控

吸濕性藥物對濕度的敏感程度，如同博物館中的古代羊皮紙文件——濕度一旦超標，損害立即且不可逆。這種情況下，監控記錄不只是行政要求，而是保存完整性的直接保障。類比到製藥：不只是監管合規，而是產品品質的最後防線。

BMS（樓宇管理系統）的 DI 關鍵要素

4b 和 4c 都使用了 BMS（Building Management System，樓宇管理系統）。BMS 的 DI 合規要求包括：

  • 系統需經過 CSV（電腦化系統驗證）
  • 稽核軌跡必須啟用且不可關閉
  • 存取控制需基於使用者角色（role-based access）
  • 電子簽名功能若使用，須符合 21 CFR Part 11
  • 資料備份與還原程序需驗證

4b 中，BMS 有這些功能但製造記錄仍是紙本，因此停留在 Moderate；4c 連製造記錄也電子化，才達到 Acceptable。

9-Box 定位：Example 4

資料關鍵性（縱軸）× 管控水準（橫軸）
─────────────────────────────────────
4a: High × Low    → Significant（紅）⚠ 需立即 CAPA
4b: High × Medium → Moderate（橙）需改善計畫
4c: High × High   → Acceptable（綠）✓ 合規目標

高關鍵性資料的合規路徑：
BMS 驗證 + 電子製造記錄 + Part 11 管理

4c 的 Part 11 要求提醒

4c 明確提到「managed per Part 11 requirements」。這代表除了技術控制，還需要相對應的程序文件：電子簽名 SOP、稽核軌跡審查程序、系統存取管理程序、以及定期的記錄完整性自我稽核。技術實施與程序文件缺一不可。

思考問題

  1. 你的 CDMO 廠房 BMS 是否已完成 CSV？最後一次 BMS 稽核軌跡審查是什麼時候？
  2. 對於吸濕性原料，你的製造現場是否有即時濕度警報系統？警報記錄如何歸檔？
  3. 若 BMS 數據與紙本手抄記錄出現差異，SOP 中的調查程序是什麼？

CDMO 應用：從 4a 升級的優先行動

  • 第一步：立即盤點哪些客戶產品具有吸濕性，對應製程環境的濕度記錄方式
  • 第二步：若目前為 4a 狀態，評估 BMS 升級或感測器直接串接的技術可行性
  • 第三步：向客戶揭露目前 DI 管控狀態，討論改善時程（客戶稽核前主動溝通優於被動發現）
  • 第四步：製訂 CAPA，以達到 4b（BMS）為短期目標，4c（EBR 整合）為長期目標

Example 4 vs 5：唯一差異在於藥物屬性

5a/5b/5c 的操作、脆弱性因素、管控措施與 4a/4b/4c 完全相同——唯一差異是藥物不具吸濕性降解風險，濕度只影響流動性而非 CQA。因此整組下移一行：Moderate → Acceptable → Negligible（相較 Example 4 的 Significant → Moderate → Acceptable）。

雖然濕度不是 CPP，但 Moderate（橙色）仍代表需要改善計畫。若客戶或稽查員詢問，「濕度不是 CPP 所以我們用紙本就好」是不完整的答案——還需要說明管控水準為何在低層級，以及何時計畫升級。

思考問題

  1. 你如何確定某個藥物的濕度影響屬於 CQA（如降解）還是製程一致性（如流動性）？這個判斷在哪份文件中記載？
  2. 若同一製程用於不同客戶的藥物，一個吸濕一個不吸濕，你的製程記錄系統是否分別設定了不同的 DI 管控要求？

CDMO 應用：多客戶 FDF 製程的 DI 差異化管理

  • 建立產品 DI 關鍵性矩陣，以藥物屬性（吸濕性、光敏性、氧敏性等）為行，以製程步驟為列，系統性填入資料關鍵性等級
  • 同一製程步驟若服務多個客戶，以最高關鍵性等級設定管控基準（向上取齊原則）
  • 每年更新此矩陣，納入新上線客戶的產品特性評估

中文解析

低關鍵性的判定：寬容濕度範圍

藥物非吸濕性且在寬廣濕度範圍內穩定，濕度既不影響 CQA 也不影響關鍵製程一致性，因此屬於低關鍵性例行 GMP 數據。這是 9-Box 最底列的典型場景。

對 CDMO 的啟示：在藥物開發早期就明確記錄此類穩定性數據，有助於為 DI 風險評估提供科學依據，而非憑直覺判斷。

6c 的警示：超額投資的反面教材

TR84 在描述 6c 時特別說明：「significantly more controls are in place than are required」。EBR + 稽核軌跡對低關鍵性資料是過度設計。這句話的監管含義是：若審計員發現你對低風險資料投入了大量資源，卻對高風險資料管控不足，這本身就是一個風險管理判斷失誤的訊號。

三組案例的整體比較

把 Examples 4、5、6 放在一起，就像同一間倉庫存放三種貨物：

  • Example 4（高）：易燃化學品——必須有完整的電子監控、即時警報、稽核軌跡
  • Example 5（中）：一般化工原料——需要基本電子記錄，紙本備份可接受
  • Example 6（低）：穩定的金屬零件——紙本入庫記錄完全足夠，不需要溫濕度監控系統

倉庫管理的原則是：根據貨物特性決定管控等級，而不是用同一套最高標準管理所有東西。

思考問題

  1. 你的 CDMO 是否有一份文件，明確列出每個客戶產品對應的製程資料關鍵性等級？若無，如何開始建立？
  2. 若現有製程中有些資料是 Significant（紅色），你有 CAPA 計畫嗎？時程是否合理？
  3. 你認為在你的 CDMO，哪一個製程步驟的 DI 管控最薄弱？它在 9-Box 的位置在哪裡？

CDMO 應用：建立 DI 風險優先順序清單

  • 第一優先：識別所有 Significant（紅色）資料——這些需要立即 CAPA，應在下次客戶稽核前完成改善
  • 第二優先：Moderate（橙色）資料——制訂 12-18 個月的改善計畫，與客戶溝通時程
  • 長期規劃：評估 EBR 平台整合的 ROI，優先整合高關鍵性製程步驟
  • 維持現狀：Acceptable + 低關鍵性的紙本記錄，確保程序受控即可，無需強制升級

三組 FDF 案例的横向比較總結

右移（升級技術）和上移（藥物更敏感）對風險等級的影響完全對稱。

Appendix I: API & FDF Worked Examples 案例學習總結

核心學習：9-Box 矩陣的實務應用邏輯

六組案例共同展示了 TR84 的核心訊息：資料完整性管控應與資料關鍵性相稱。高關鍵性資料需要高管控（EBR + 稽核軌跡），而低關鍵性資料以紙本加人工審查就足夠。

  • API 案例（Ex1-3）：從反應控制（高）→ 烘乾（中）→ 批次間清潔（低），展示相同技術路徑（紙本 → PLC → EBR）在不同關鍵性下產生不同的脆弱性結果。
  • FDF 案例（Ex4-6）：從吸濕性藥物（高）→ 影響流動性（中）→ 穩定藥物（低），展示相同製程在不同產品屬性下的風險差異。
  • 關鍵洞察：高關鍵性資料無法達到 Negligible；低關鍵性資料無需達到 Negligible。這是「合理合規」而非「最高標準」的體現。

CDMO 稽核就緒的三個關鍵問題

  1. 你的每個製程步驟，資料關鍵性等級是否已書面評估？（有依據的科學判斷，而非假設）
  2. 現有管控水準是否與關鍵性相稱？（不能只有最低管控卻有高關鍵性資料）
  3. 若存在 Significant 或 Moderate 風險，CAPA 計畫是否有時程和負責人？（有問題不可怕，無行動才是問題）

最終比喻：DI 管控是安全帶，不是防彈玻璃

TR84 Appendix I 的案例告訴我們：數據完整性管控不是「越多越好」，而是「恰到好處」。開車時繫安全帶是必要的；但要求每輛家用車都安裝賽車防滾架，既不現實也不必要。製藥 DI 管控的藝術，在於準確識別哪些是「高速賽車」（高關鍵性資料），為它們配備完整防護；並認識到哪些是「市區代步車」（低關鍵性資料），維持基本安全即可。
