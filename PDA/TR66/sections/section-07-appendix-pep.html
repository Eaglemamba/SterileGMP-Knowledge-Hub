9.1 Introduction

9.1.1 Purpose

This appendix contains an example project execution plan (PEP). This material is referenced in Section 7.4.1, Project Execution Plan, and should be used as an example for how this type of document is structured. The complexity and size of the document will change according to the type of SUS and complexity of the project. These types of documents should be considered as living (i.e., they can be updated as appropriate should the project change). The document can be prepared by the end user or by a sponsor of the end user such as an engineering company or supplier.

The purpose of this PEP is to define and describe the project execution strategy as well as how the project will be managed and how quality control is applied in order to ensure that the project meets its goals regarding cost, time, and quality.

9.1.2 Context of this Document

This document provides a road map for the work to be completed. The PEP is a "fit-for-purpose" and living document that defines the supervising engineering contractor's (SEC's) scope of services and responsibilities for the project.

This plan describes and defines the project scope, the execution strategy, quality control systems, and management plan.

9.1.3 Revision of the PEP

All information included in this PEP is current as of the latest issue date of each document referred to. If changes are made that significantly affect the execution strategy, project management, and/or quality control activities as defined herein, a revision shall be made with rationale for the change explained in the new revision. The approval of the new revisions follows the same procedure as the approval of the original PEP. If there is a change to an appendix or a referred document to this PEP, only the document will be revised.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

Project Execution Plan / PEP (專案執行計畫)：PEP 是整個 SUS 建廠或改造專案的「作戰計畫書」，涵蓋範圍定義、執行策略、品質管控與時程管理。它不是一份寫了就束之高閣的文件，而是一份「活的文件 (Living Document)」，隨專案進展持續更新。

Supervising Engineering Contractor / SEC (監督工程承包商)：在大型製藥工程專案中，業主通常委託 SEC 來統籌設計、採購、施工與確效等工作。SEC 扮演專案管理的核心角色。

比喻說明 Analogy

PEP 就像蓋房子時的「施工藍圖+工程管理計畫」。想像你要裝修一間新餐廳：你需要一份文件說明「要做什麼（範圍）」、「怎麼做（策略）」、「誰負責什麼（組織）」、「品質怎麼控管」、「時程怎麼安排」。PEP 就是把這些全部整合在一份主文件中，讓所有利害關係人有共同的依據。

重點提示 Key Notes

PEP 的「活文件」特性非常重要。在 SUS 專案中，由於一次性技術的供應商生態系仍在快速演進，專案執行過程中經常需要調整設計或供應商選擇。PEP 必須有明確的版本管控 (Revision Control) 機制，確保每次變更都經過正式審核與批准。

請注意：PEP 可以由最終使用者 (End User) 或其委託的工程公司/供應商來撰寫，但無論誰撰寫，業主都必須審核並批准。

9.2 Project Overview

9.2.1 General

Company X wishes to design and equip a new production suite, GMPX, for manufacture of injectable products for viral particle gene therapy.

The products manufactured in the new production suite shall be based on modified live viral particle technologies (gene therapies). The manufacturing process shall cover production of biological bulk and final filled products for sterile, liquid drug applications. The viral developments are based on adenovirus. The production portfolio covers in the first place two bulk and two final products, which are known as Product 1 and Product 2.

9.2.2 Client

The client for the GMPX project is Company X.

9.2.3 Project History

A basic design was earlier developed for a traditional non-SUS approach.

The building for GMPX is already constructed alongside building T4, and all basic utility installations are available there.

9.2.4 Project Scope

The overall purpose of the project is:

  • To design and equip a new manufacturing suite, to be integrated into the T4 complex, for manufacture of injectable products for viral particle gene therapy with disposable technology
  • To finalize OQ for the manufacturing suite within 18 months from start of basic design within agreed project budget

9.2.5 Project Objectives, Success Criteria, and Focus Areas

(Refer to the original document for the detailed table of objectives, success criteria, and focus areas.)

9.2.6 Steering Committee

The SEC and Company X will implement a common steering committee for the project. The purpose and responsibilities of the steering committee are to:

  • Assure that the project goals are met
  • Create a common picture of questions, problems, and limitations within each organization
  • Create a common picture of roles and responsibilities between parties

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

Gene Therapy (基因治療) 與 SUS 的關聯：此範例非常典型地展示了為什麼 SUS 特別適合基因治療產品。病毒載體 (Viral Vector) 製造涉及活病毒操作，需要高生物安全等級 (BSL) 的環境。使用一次性系統可以大幅降低交叉污染風險，並簡化清潔驗證的複雜度。

Steering Committee (指導委員會)：由 SEC 與業主共同組成的最高決策機構，負責監督專案進度、審批計畫變更、管控預算與時程。

比喻說明 Analogy

把這個專案想像成「從傳統瓷器餐廳改造成使用免洗餐具的外送廚房」。Company X 原本規劃了傳統不鏽鋼 (non-SUS) 的產線，但後來決定改用一次性技術。建築已經蓋好了（就像廚房空間已經有了），現在要做的是把裡面的設備全部用一次性系統來重新設計和安裝。

重點提示 Key Notes

請特別注意專案時程目標：從 Basic Design 開始到 OQ 完成，僅有 18 個月。這在傳統不鏽鋼產線幾乎不可能達成，但 SUS 因為省去了大量管路安裝、焊接檢查與 CIP/SIP 驗證，大幅縮短了建廠時程。這正是 SUS 的核心商業優勢之一。

Steering Committee 的設置也值得學習：它確保了 SEC 與業主之間有結構化的溝通管道，避免因資訊不對稱而導致的專案延誤或品質問題。

實務應用 Practical Application

作為 CDMO，如果你要為客戶建置一條基因治療的 SUS 產線，這份 PEP 範例提供了非常好的結構參考。你可以直接套用其範圍定義、組織架構和品質策略的框架，再根據自身需求調整細節。特別是 Steering Committee 的設計，在 CDMO 與 Sponsor 之間的協作中尤為重要。

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

Contract Documents (合約文件)：這裡區分了 Basic Design（基本設計）和 Detailed Design and Execution（細部設計與執行）兩個階段的合約文件。這種分階段簽約的方式在大型製藥工程中很常見，可以降低業主的風險。

User Requirement / UR (使用者需求)：由業主提供的需求規格書，是整個設計的基礎。在 SUS 專案中，UR 需要特別涵蓋一次性組件的材質相容性、萃取物/浸出物規格等特殊要求。

比喻說明 Analogy

合約文件就像婚前協議書：在專案「結婚」之前，雙方必須清楚約定各自的責任、交付物和權利義務。Basic Design 階段的合約就像「訂婚」，先確認大方向；Detailed Design 階段的合約則是「正式結婚」，把所有細節都寫清楚。

重點提示 Key Notes

注意 Customer X 的交付責任中包括「Process know-how and product requirements」。這在 SUS 專案中特別重要，因為一次性組件的選擇（如袋材、管路材質）必須基於製程需求和產品特性。業主如果無法及時提供這些資訊，將直接影響 SEC 的設計工作。

9.4 Scope

9.4.1 Objectives of the Scope

The GMPX project activities such as value engineering and thorough schedule control of all design, construction, commissioning and qualification (C&Q), and changes will have a high importance throughout the project.

9.4.2 Scope Supervision

The scope and progress will be supervised by the steering committee. Changes in the scope will be handled by change orders as described in Section 4.6.

9.4.3 Scope of Engineering

All engineering required to supply and build the process building is within the scope of the SEC. The scope comprises design development of the technical documents prepared during basic design to a detailed design status (i.e., the status required for manufacturing, installation, commissioning, operation, and maintenance of the facility), with all the installed equipment.

The basic design documents have been reviewed and approved by Company X and constitute the project baseline, setting the specific requirements for the project. Any changes from this baseline are to be taken up through the change management procedure.

Reference documents (including client inputs) used during basic design become backup documents. In case of contradiction between requirements specified in the basic design documents and requirements specified in any of the reference documents, then the latter shall take precedence, unless such requirement is superseded by a subsequent change management.

During detailed design the information set by the basic design package is used to develop construction documentation.

9.4.3.1 Design Steps Overview

The SEC will provide a complete design of the new manufacturing suite in the following steps:

  • Front-end definition
  • Basic design
  • Detailed design

9.4.3.2 Basic Design

The SEC will prepare:

  • Project proposal with executive summary project contents
  • Project plan
  • Biosafety level impact assessment

9.4.3.3 Detailed Design

The SEC will provide engineering with:

  • Approved and signed piping and instrumentation diagrams
  • Approved building layout
  • Approved main documents such as UR, PEP, and QAP

9.4.4 Scope of Equipment and Systems

9.4.4.1 General

The scope includes all architectural considerations for cleanroom and mechanical spaces (with exception of the outer building walls) and finishes, building installations for HVAC, utility piping, safety, process piping, process equipment, electrical, and automation.

9.4.4.2 Spare Parts

The SEC will provide spare parts required for commissioning, testing, and qualification.

The SEC will also provide critical spare parts for two years from installation date.

9.4.4.3 Consumables

The SEC will supply the consumables, such as filters, gaskets, lubricant, lamps and lights, fluorescents, and bags, that are required until end of commissioning. Subsequent consumables shall be provided by Company X.

9.4.4.4 Disposables

Disposable bags for cell expansion will be supplied by the SEC based on the UR for the bags in a limited amount only if necessary for IQ and OQ purposes.

9.4.4.5 Utilities

The running cost for utilities during construction and commissioning and qualification (C&Q) will be accounted directly to Company X. All temporary utilities needed only during construction and C&Q will be supported by the project.

9.4.5 Change Management

Management of project change is essential to successful project execution. The project will implement a change management process that will provide for timely identification of deviations or changes in project scope.

All change orders shall be recorded in a change-order log.

See QAP for change control of documents and C&Q activities.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

Value Engineering (價值工程)：在不降低功能和品質的前提下，尋找降低成本的替代方案。在 SUS 專案中，這可能包括評估不同供應商的一次性組件、優化管路配置以減少接頭數量等。

Project Baseline (專案基線)：經過批准的 Basic Design 文件構成專案基線。任何偏離基線的變更都必須走正式的變更管理程序。這是防止 Scope Creep（範圍蔓延）的關鍵機制。

Design Steps（設計步驟）：Front-end Definition → Basic Design → Detailed Design 是標準的三階段設計流程。每個階段的細緻程度逐步增加，也代表著投入的資源和承諾逐步加深。

比喻說明 Analogy

設計三階段就像畫一幅油畫：Front-end Definition 是構圖（確定要畫什麼）、Basic Design 是素描底稿（確定比例和結構）、Detailed Design 是上色完稿（所有細節都到位）。如果底稿畫錯了，上色階段再改就非常昂貴。這就是為什麼 Basic Design 階段的審核和批准如此重要。

重點提示 Key Notes

SUS 專案中 Disposables 的特殊考量：注意 9.4.4.4 提到一次性袋子「僅在 IQ 和 OQ 必要時提供有限數量」。這是因為一次性組件有保質期限制，如果太早大量採購，可能在 PQ 或正式生產前就過期了。SUS 專案的採購策略必須考慮組件的 shelf life。

文件優先順序原則：當 Basic Design 文件與參考文件產生矛盾時，參考文件優先（除非被後續變更管理所取代）。這個原則確保了業主的原始需求不會被設計過程中的解讀所稀釋。

  • Authority review and approval before any construction activity starts
  • Occupancy permits (if applicable); need to be applied for by Company X

9.5.2.7 As-Built Drawings

Approved documents that need to be as-built have to be identified in a later stage.

These documents will be redlined at the site during construction and C&Q.

Redlined documents can be used as as-built documents for C&Q and handover of the systems. The work will be a joint effort of the SEC, vendors, and, if applicable, Company X.

These documents will thereafter be reissued (CAD) as revisions as-built by the SEC.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

Concurrent Engineering (並行工程)：PEP 明確指出 DD、施工和 C&Q 將會重疊進行。這是「快速建廠 (Fast-Track Construction)」的核心策略，透過並行作業來壓縮整體時程。

As-Built Drawings (竣工圖)：施工過程中，實際安裝往往與原始設計有偏差。Redline（紅線標記）就是在現場直接在原圖上標記實際差異，最終由 SEC 正式重新出圖成為 As-Built 版本。

Design Disciplines（設計專業）：將工程拆分成建築、製程技術、公用設施和自動化四大領域，每個領域下再細分。這確保了每個專業都有對應的負責人 (Design Lead)。

比喻說明 Analogy

並行工程就像邊蓋房子邊裝潢：一樓還在砌牆的時候，二樓的設計圖已經在畫了。這種方式可以大幅縮短時程，但要求各團隊之間的溝通協調必須非常緊密，否則「一樓牆壁的管線預留孔」可能跟「二樓的管路設計」對不上。

重點提示 Key Notes

QA 獨立性：注意 9.5.2.2 特別強調 QA 在 DD 階段是以「獨立角色」參與審查 GMP 合規性。這符合 GMP 的基本原則——品質保證功能必須獨立於生產和工程，避免利益衝突。

文件分級審批：並非所有設計文件都需要業主審批。僅「Governing Design Documents」（主要設計文件）才送交業主。純施工用的詳細圖紙由 SEC 內部管控。這是務實的做法，避免業主被大量文件淹沒。

圖表解讀 Figure Analysis

專案階段流程圖清楚顯示了從 DD 到 Handover 的完整生命週期。關鍵里程碑包括：

  • Documented Mechanical Completion（文件化機械完工）：施工完成的正式確認點
  • ATO (Approval to Operate)：批准運轉的關鍵門檻，代表系統已通過 IQ/OQ
  • CCC (Care, Custody, and Control)：系統從 SEC 移交給業主管理的轉折點

9.5.4 Commissioning and Qualification

Execution of C&Q is described in the commissioning and qualification plan GMPX-C&Q-07-001 and in the QAP Doc. no. GMPX-QAP-07-001. The execution can be summarized as follows.

9.5.4.1 Factory Acceptance Test (FAT) and Site Acceptance Test (SAT) for Systems Assessed as Indirect Impact and Direct Impact

Suppliers and contractors of indirect- and direct-impact systems must, if applicable according to SQP during FAT/SAT, demonstrate to the SEC that all requirements, including functions' "as-built" documentation, are fulfilled and available and that the deliverables operate as required. In situations where the supplier or contractor is not in a position to perform FAT/SAT as expected, the SEC, in cooperation with the supplier or contractor, must perform additional tests, possibly as commissioning.

FAT/SAT shall be prepared in a manner that ensures that all C&Q requirements are tested and documented and fulfill the acceptance criteria at the completion of SAT.

When the machine or product has been assembled at the site and adequately connected and run-in tested by the supplier or contractor, the SAT is carried out. This takes place in a way similar to the FAT, according to SEC-approved SAT protocols. For indirect-impact and direct-impact systems, SEC QA should review protocols and test plans in the planning of SAT to ensure that approval of the SAT report is obtainable. The SAT is finished with the SEC and SEC QA approval of the SAT report.

9.5.4.2 Commissioning Indirect-Impact Systems and Direct-Impact Systems

For systems assessed as direct or indirect impact, fulfillment of C-requirements shall be demonstrated through commissioning. The work package owner (WPO) has the responsibility to ensure that commissioning activities are well planned and adequately executed, documented, and reported. The Q-coordinator is responsible for overall coordination and follow-up.

Vendor SAT is part of the commissioning.

9.5.4.3 No-Impact Systems

For systems assessed as having no impact, fulfillment of requirements must be demonstrated and documented according to GEP based on the tradition in the discipline.

The WPO is responsible for planning and executing the commissioning activities for no-impact systems and to ensure that the activities are documented in a way that allows the SEC to illustrate that all requirements are met.

9.5.4.4 Qualification

For systems assessed as direct impact, fulfillment of Q-requirements in relation to installations and operations shall be demonstrated through installation qualification and operational qualification (IQ/OQ). Qualification and especially prerequisites before the qualification can start shall be described in the protocols. IQ and OQ must be carried out according to the approved protocols.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

Impact Assessment（影響評估）：系統被分為三類——Direct Impact（直接影響）、Indirect Impact（間接影響）和 No Impact（無影響）。這個分類決定了 C&Q 的深度與嚴格程度：

  • Direct Impact：直接影響產品品質的系統（如生物反應器、充填設備），需要完整的 IQ/OQ
  • Indirect Impact：間接影響的系統（如 HVAC、純水系統），需要 FAT/SAT + Commissioning
  • No Impact：不影響產品品質的系統（如照明），僅需按 GEP 標準進行調試

C-requirements vs Q-requirements：C-requirements 是透過 Commissioning 驗證的工程需求，Q-requirements 是透過正式 Qualification (IQ/OQ) 驗證的 GMP 需求。這種區分是 ASTM E2500 和 ISPE 推薦的基於科學和風險的確效方法。

比喻說明 Analogy

Impact Assessment 就像醫院的感染風險分級：手術室（Direct Impact）需要最嚴格的消毒和監控、病房走廊（Indirect Impact）需要基本的清潔標準、停車場（No Impact）只需一般維護。每個區域的管控強度與其對患者安全的影響成正比。

重點提示 Key Notes

FAT/SAT 的 SUS 特殊性：在 SUS 專案中，FAT 通常在一次性組件供應商的工廠進行，重點驗證組件的品質管控系統和製造一致性；SAT 則在使用者現場驗證組件在實際製程條件下的表現。

採購以業主名義下單：雖然 SEC 負責採購，但訂單以 Company X 的名義下達。這確保了業主對供應商關係的最終控制權，也有利於保修和售後服務的銜接。

實務應用 Practical Application

在 CDMO 的 SUS 專案中，Impact Assessment 的結果直接決定了確效工作量和預算。一個常見的錯誤是將太多系統分類為 Direct Impact，導致確效成本膨脹。務實的做法是：嚴格按照系統對產品品質的實際影響來分類，並有科學依據支持分類決定。例如，SUS 的混合袋如果僅用於緩衝液配製，可以考慮分類為 Indirect Impact 而非 Direct Impact。

練習思考 Practice Questions

  1. 在一個 SUS 生物反應器系統中，哪些組件屬於 Direct Impact？哪些可能是 Indirect Impact？
  2. 如果供應商無法在工廠進行 FAT，SEC 應該如何處理？這對專案時程有什麼影響？
  3. C-requirements 和 Q-requirements 的區分對確效效率有什麼實際好處？

9.7 Quality

9.7.1 Project Quality Strategy

A validation master plan (VMP) is produced by Company X for the project. This VMP will be used to provide the basis for all validation and quality activities during the lifecycle of the project.

The SEC is responsible for carrying out the commissioning and qualification activities (IQ and OQ) unless otherwise described in the VMP. The subsequent process qualification, cleaning validation, and process validation are carried out by Company X and are not included in this project.

The VMP is the overall document for validation in the project and will be master and basis for qualification of equipment facilities and utilities. A separate VMP will be written for process validation.

The qualifications sequence of CP, IQ, OQ, and PQ activities are described in details in the QAP.

9.7.1.5 Partner/Vendor/Supplier Qualifications

Partners and suppliers describe their "normal" test in the tender documents (for instance, via examples of test plans). If possible, the tests are performed on the GMP level when relevant.

  • Tests must address all requirements.
  • Scope of the test activity is determined in the contract (the partners and suppliers draft QAP).
  • After this, the SEC prepares a draft of IQ and OQ protocol in which test plans from the supplier may be entered, if relevant.
  • The suppliers test planning may be adjusted to achieve maximum synergy.
  • The SEC approves supplier's final QAP.

Time schedule and economy of the project includes a completed IQ and OQ with the SEC as the final step. The validation involves the active participation of the end user and Company X-QA. The scope of this is described in the basic design phase.

9.7.2 Quality Activity Plan (QAP)

The project's QAP manages all quality activities and quality documentation to be prepared. The QAP is made in compliance with the requirements in the VMP.

9.7.3 Document Control

Document control is handled in accordance with the QAP.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

VMP (Validation Master Plan / 驗證主計畫)：VMP 是整個驗證活動的最高層級文件，由業主 (Company X) 負責撰寫。它定義了驗證的範圍、策略、角色分工和接受標準。在此專案中，VMP 涵蓋設備確效 (IQ/OQ)，但製程驗證 (PV) 另立專屬 VMP。

QAP (Quality Activity Plan / 品質活動計畫)：QAP 是 VMP 的執行層級文件，詳細列出每項品質活動、負責人、時程和交付物。可以把 VMP 視為「品質的憲法」，QAP 則是「品質的執行辦法」。

Supplier Qualification Synergy（供應商確效整合）：PEP 提出了一個非常務實的做法：將供應商的標準測試計畫整合到 SEC 的 IQ/OQ 方案中，避免重複測試。這在 SUS 專案中尤其重要，因為一次性組件供應商通常已經有完善的出廠測試程序。

比喻說明 Analogy

VMP、QAP 和 IQ/OQ Protocol 的關係就像法律體系：VMP 是「憲法」（最高指導原則）、QAP 是「法律」（具體規範）、各個 IQ/OQ Protocol 是「作業標準程序」（具體執行步驟）。每一層都必須符合上一層的要求，不能互相矛盾。

重點提示 Key Notes

SUS 專案中清潔驗證的簡化：注意 PEP 提到「cleaning validation」由 Company X 負責且不在此專案範圍內。在 SUS 設施中，由於使用一次性接液面組件，傳統意義上的清潔驗證大幅減少，但仍可能需要針對可重複使用的設備外部和環境進行清潔確效。

確效分工的清晰界定：SEC 負責到 OQ，Company X 從 PQ 開始接手。這個交接點必須非常明確，因為 OQ 的完成品質直接影響 PQ 能否順利進行。

  2. 將供應商測試計畫整合到 IQ/OQ 方案中有什麼好處？可能有什麼風險？
  3. 在 SUS 專案中，清潔驗證的範圍與傳統不鏽鋼設施相比有何不同？

9.8 Risk Management

9.8 Risk Management

There will be a number of risk analyses performed during the project. Their main purpose is to identify risk early and create activities to reduce the risk.

The risk analyses are:

  • BD Project Risk Analysis 1
  • BD BSL and GMP Risk Analysis with Focus on the Layout
  • DD Project Risk Analysis 2
  • DD HSE Risk Analysis on the Execution Phase

9.8.1 Documentation

  • Language and templates — Project language is UK English with the exception of documents used for Finnish craftsmen. The SEC templates are used when adequate.
  • Organizing project data — Data storage must be planned for and organized at the start of the project in order to properly archive the information during project closeout. Proper setup at the start of the project will help make it possible to utilize data efficiently, minimize rework, and ensure complete electronic records retention.
  • Project documentation — In basic design the planned documents are described in the object breakdown structure. The quality managed documents are described in the QAP. The document delivery of the suppliers and partners are described in the tender documents or in the contract of the project.
  • Technical tools and standards — For each skill or profession (construction, architecture, bulk process, utility, etc.) it is necessary to determine and develop tools and standards to ensure continuous engineering of the project parts. The lead engineers manage these tools and standards. Specification tools and standards are being prepared in basic design.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

分階段風險分析：PEP 規劃了四次風險分析，分別在 Basic Design (BD) 和 Detailed Design (DD) 階段進行。這體現了「漸進式風險管理」的原則——隨著設計細節的增加，風險分析的焦點也從宏觀（專案風險、佈局風險）轉向微觀（執行風險、HSE 風險）。

BSL (Biosafety Level / 生物安全等級) Risk Analysis：由於本專案涉及活病毒操作，BSL 風險分析特別重要。它需要評估病毒逃逸風險、人員暴露風險以及環境釋放風險，並確保設施設計能夠有效控制這些風險。

比喻說明 Analogy

分階段風險分析就像飛行員的「檢查清單」：起飛前有一份（BD Risk Analysis）、巡航時有一份（DD Risk Analysis）、準備降落時還有一份（HSE on Execution）。每個階段面對的風險不同，需要不同的檢查重點。不是做一次就夠了，而是在每個關鍵節點重新評估。

重點提示 Key Notes

電子記錄管理的前瞻性：9.8.1 強調在專案開始時就要規劃資料存儲。這在現代 GMP 環境中尤其重要，因為法規要求（如 21 CFR Part 11 和 EU GMP Annex 11）對電子記錄的完整性、可追溯性和長期保存有嚴格要求。專案關閉時如果發現文件缺失或格式不兼容，補救成本將非常高昂。

  2. 為什麼要在 Basic Design 階段就進行 BSL 和 GMP 風險分析？如果推遲到 Detailed Design 階段，可能會發生什麼？

9.9 Health, Safety, and Environment

An initial HSE evaluation has been started. The first version of this plan will be issued and approved before any construction activity starts. This plan will be updated during the construction with new items and activities.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

HSE (Health, Safety, and Environment / 健康、安全與環境)：HSE 計畫是法規強制要求的文件，在施工開始前必須批准。在涉及活病毒操作的 SUS 設施中，HSE 還需要特別考慮生物安全相關的風險，如病毒氣溶膠暴露、廢棄一次性組件的滅活處理等。

9.10 Commissioning and Qualification

9.10 Commissioning and Qualification

When the SEC finds it beneficial, an all-day meeting will be called in order to approve documents and/or carry out reviews. This will be scheduled no less than one week in advance and with duration from one day to several days.

When the SEC finds it beneficial, an all-day commissioning day will be called in order to correct errors and carry through cleanroom documents. This will be arranged no less than one week ahead and with duration from one day to several days.

The SEC is responsible for carrying out or supervising and reporting all qualification activities up to and including OQ.

Company X will participate in the test activities.

Company X provides QA resources during the OQ test period.

Company X is responsible for the calibration and maintenance after OQ.

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

C&Q 交接點：此節清楚劃定了 SEC 與 Company X 在確效活動中的分工界線。SEC 的責任止於 OQ 完成；OQ 之後的校正、維護和 PQ 全部由 Company X 接手。這是業界標準的交接模式。

比喻說明 Analogy

這就像買新車的交車流程：經銷商（SEC）負責到把車開到你面前、完成所有檢查（IQ/OQ）；交車之後的保養、加油和日常維護（PQ、校正、維護）就是車主（Company X）的責任了。但交車時雙方都要在場確認車況（Company X 的 QA 資源參與 OQ）。

重點提示 Key Notes

OQ 後的校正與維護責任轉移：這個細節很容易被忽略。Company X 必須在 OQ 完成前就建立起內部的校正和維護能力（包括 SOP、人員培訓和備件庫存），否則在 SEC 撤出後可能出現管理真空期。

9.11 Handover and Operation Support

9.11 Handover and Operation Support

The project is handed over to the customer in three steps: 1) ready to run, 2) handover, and 3) final handover. Final handover is the last part of the project.

The project is handed over to the customer with the participation of the SEC project manager, relevant SEC and/or contract project engineers, and representatives from the customer of his or her own choosing.

If the customer wishes to put the area into use before the handover has taken place, the SEC project manager shall ensure that a physical examination of the area takes place together with the customer or the user and that an agreement on operation and maintenance responsibilities is made.

Prior to final handover, deviation lists will be resolved by a cleanup team.

Warranties from third-party vendors, suppliers, and subcontractors are handed over to Company X.

PQ and PV activities are not part of this project. The SEC recommends the client to plan on a PQ support team. SEC will supply resources for support in PQ/PV phase if Company X so wishes. The SEC resources will work under Company X supervision and management during PQ/PV.

Company X has full responsibility for running and maintaining equipment after handover. This must be carried out without invoking any delays to the project.

Documentation will be handed over in binders and/or electronic copy as specified by Company X at final handover, and direct to Company X document control. Where the SEC has produced or obtained from a supplier an electronic copy, this will be provided with the final documentation.

Optimizing program and parameters is not within the scope of this project. Education and training are to be defined. (See C&Q plan.)

導師解析 Tutorial Commentary

核心概念解析 Key Concepts

三階段移交 (Three-Step Handover)：

  • Ready to Run（就緒運轉）：設備已安裝完成、通過 OQ，可以開始運轉
  • Handover（移交）：正式將系統的操作與維護責任從 SEC 轉移給 Company X
  • Final Handover（最終移交）：所有偏差處理完畢、文件齊全、保固轉移，專案正式結束

Deviation Lists / Punch Lists（偏差清單/缺陷清單）：在移交前，會有一份列出所有待解決問題的清單。Cleanup Team 負責在最終移交前逐一解決這些問題。

比喻說明 Analogy

三階段移交就像新房交屋：第一階段「預驗收」（Ready to Run），確認硬體完工；第二階段「交鑰匙」（Handover），你可以開始搬進去住了；第三階段「尾款結清」（Final Handover），所有瑕疵修繕完畢、保固書交接、裝修公司撤場。每一階段都需要雙方簽字確認。

重點提示 Key Notes

PQ/PV 的銜接規劃：PEP 明確指出 PQ/PV 不在此專案範圍內，但 SEC 建議業主提前組建 PQ 支援團隊。這是非常務實的建議——許多專案在 OQ 完成後才發現沒有足夠的內部能力執行 PQ，導致產品上市時間延遲。CDMO 在承接 SUS 專案時，應在 PEP 階段就將 PQ/PV 的人力和資源需求納入規劃。

提前使用的風險管控：如果業主在正式移交前就需要使用設施，PEP 要求進行現場檢查並簽訂操作和維護責任協議。這是為了釐清責任界線，避免在出現問題時產生歸責爭議。

實務應用 Practical Application

對 CDMO 而言，Handover 階段是專案成功與否的關鍵時刻。建議在 PEP 中加入「知識轉移計畫 (Knowledge Transfer Plan)」，確保 SEC 的技術團隊在撤出前將所有操作知識和維護經驗傳授給 CDMO 的運營團隊。尤其是 SUS 專案中一次性組件的安裝技巧、無菌連接操作和故障排除經驗，這些「隱性知識」往往比書面文件更有價值。

特別注意以下縮寫在 SUS 專案中的特殊意義：

  • GEP (Good Engineering Practice / 良好工程實務)：適用於 No-Impact 系統的驗證標準
  • BSL (Biosafety Level / 生物安全等級)：在基因治療產品製造中是關鍵設計參數
  • SEC (Supervising Engineering Contractor / 監督工程承包商)：此範例中的核心執行角色

重點提示 Key Notes

GEP 和 GMP 的區分是理解 C&Q 策略的關鍵：GEP 適用於不直接影響產品品質的系統，要求較低但仍需文件化；GMP 適用於直接影響產品品質的系統，要求正式的確效文件和審批流程。在 SUS 專案中，正確區分兩者可以顯著降低確效工作量和成本。

核心概念解析 Key Concepts

PEP 的配套文件體系：PEP 不是一份孤立的文件，它與多份配套文件共同構成完整的專案管理框架：

  • Level A Time Schedule（主時程表）：專案的最高層級時程，標示關鍵里程碑
  • QAP：品質活動的詳細計畫
  • VMP：驗證策略的最高指導文件
  • Object Breakdown Structure（設備分解結構）：將設施拆解為可管理的單元
  • Procurement Strategy（採購策略）：定義供應商選擇和採購流程

重點提示 Key Notes

在 SUS 專案中，Procurement Strategy 特別重要，因為一次性組件的供應商選擇直接影響製程設計和產品品質。採購策略需要考慮：供應商的法規合規狀態、組件的 E&L (Extractables & Leachables) 數據可用性、供應鏈穩定性（單一供應商風險）以及交貨前置時間。

本節重點回顧 Section Summary

  • PEP 是 SUS 專案的核心管理文件：它定義了專案範圍、執行策略、組織架構、品質管控和時程規劃，是所有利害關係人的共同依據
  • 活文件特性：PEP 隨專案進展持續更新，每次重大變更需經正式審核與批准程序
  • Impact Assessment 決定 C&Q 深度：Direct Impact 系統需完整 IQ/OQ，Indirect Impact 需 FAT/SAT + Commissioning，No Impact 僅需 GEP 級調試
  • SEC 與業主的分工界線：SEC 負責設計、施工、採購和確效（至 OQ），業主負責 VMP 制定、PQ/PV 執行和後續運維
  • 三階段移交機制：Ready to Run → Handover → Final Handover，確保責任有序轉移、文件完整交接
  • SUS 專案的特殊考量：一次性組件的 shelf life 管理、E&L 數據需求、供應商單一來源風險，以及清潔驗證範圍的簡化
  • 品質文件層級：VMP（憲法）→ QAP（法律）→ IQ/OQ Protocols（SOP），逐級承接且不得矛盾
  • 風險管理貫穿全程：從 BD 到 DD 分階段進行風險分析，涵蓋專案風險、GMP/BSL 風險和 HSE 風險
