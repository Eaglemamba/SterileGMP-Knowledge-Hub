6.3.8 Results Printing or Publishing 6.3.8 結果列印或發布

原文內容 Original Text

Some software packages use the term "printing" to generate a final result; in this case, printing does not mean output to paper form. Printing, or "publishing," refers to the next step in generating output once testing is complete. In a well-configured system, the results after this step are final and may not be changed without reprocessing. Reprocessing must be governed by procedures and software that is correctly configured and should require a comment or justification. Before publishing, the supervisor or Quality Unit should review the results, according to established laboratory procedures. Though printing results on paper is not an ideal practice, some firms still use this approach, often due to limited software functionality or limited resources.

The following items or reports should be generated, through publishing or software functionality, after each analysis and made available in the system for review:

  • Sample sequence
  • Instrument method
  • Integration events/processing method
  • Results (chromatograms)
  • Results audit trail (if available)
  • Sequence audit trail (if available)
  • Method audit trail (if available)

The results should include the following criteria:

  • Analyst's name and signature
  • Date and time results were processed
  • Results audit trail, method audit trail, and sequence audit trail
  • Chromatograms presented in visible scale per respective analysis (peak top for assay or single-analyte analysis, peak base for purity analysis)

導師解析 Tutorial Commentary

核心概念：Publishing ≠ 列印到紙

「Publishing」是 ALCS 術語，指將測試結果鎖定並正式輸出的動作——無論是電子發布或紙本列印。一旦 Publish，結果即為最終資料（final data），不可在不留痕跡的情況下修改。

若需修改已發布結果，必須走Reprocessing（再處理）流程，並強制要求填寫修改理由（justification）——這正是資料完整性設計的核心防線。

類比：Publishing 如同公文用印

把分析結果想像成一份公文：
• 分析中 = 草稿階段，可自由修改
• Publishing = 主管蓋章用印，正式生效
• 用印後修改 = 必須收回、重新簽核，並記錄原因

ALCS 的 Publishing 機制正是數位版的「用印管控」。

實務注意：紙本列印的風險

TR80 明確指出：紙本列印不是理想做法，但仍有廠商因軟體限制或資源不足而使用。若採用紙本，審查員需確認：

  • 原始電子資料是否仍可存取並為最終版本
  • 紙本是否有審查者簽名及日期戳記
  • 再列印（re-printing）是否有管控程序，防止選擇性報告

查核準備：必須產出的七項報告

每次分析後，系統中必須可取得以下報告供審查：

  1. 樣品序列（Sample sequence）
  2. 儀器方法（Instrument method）
  3. 積分事件/處理方法（Integration events）
  4. 結果色譜圖（Results/chromatograms）
  5. 結果稽核軌跡（Results audit trail）
  6. 序列稽核軌跡（Sequence audit trail）
  7. 方法稽核軌跡（Method audit trail）

查核官會要求調出上述所有報告——任何一項缺失都是缺陷的潛在來源。

6.3.9 Audit Trails — Overview 6.3.9 稽核軌跡 — 概述

原文內容 Original Text

An audit trail is a chronology of the "who, what, when, and why" of a record. It ensures a secure, computer-generated, time-stamped electronic record is available that allows for reconstruction of the course of events relating to the creation, modification, or deletion of an electronic record. For example, the audit trail for an HPLC run could include the user name, date and time of the run, integration parameters used, and details of reprocessing, if any, including the change justification. In addition, as part of reconciliation, the injection log can be included, but this injection log does not include audit trail injection information (blank/system suitability/Lot#).

The audit trail feature (or function) should be enabled at all times, with strict access controls in place; equipment with data processing capability but lacking audit trail capability should be upgraded to CGMP standards. In those instances where the equipment lacks such capability, appropriate controls (e.g., paper-based) should be implemented to ensure all data and information generated, along with the history of all activities performed, is available for review.

The audit trail should include all possible data changes; then, the reviewer can decide whether any data change represents a compromised or regulated event.

Different types of audit trails supply data on different functions; each should be assessed individually and, then, all audit trails should be assessed collectively to determine the integrity of the data. Instrument-specific software provides several fields for configuring audit trails, and templates from which to choose for presentation. Figure 6.3.9-1 illustrates one approach to audit trail classification.

The Quality Unit is responsible for enabling fully functional audit trails, assessing the different kinds of metadata captured, and choosing which fields are required to verify and ensure data reliability and compliance. This approach should be carefully selected with a view to design the most appropriate procedures for batch release as well as a sound periodic review.

The following list reflects one recommended approach to the various classifications of audits:

  • System-level audit trail
  • Application-level audit trail
  • Method audit trail
  • Results audit trail (or injection audit trail)
  • Sequence audit trail
  • Transaction log/system error log

核心定義：稽核追蹤的四大要素

稽核追蹤（Audit Trail）必須回答四個問題：

  • WHO（誰）— 使用者身份
  • WHAT（做了什麼）— 操作內容
  • WHEN（何時）— 時間戳記
  • WHY（為什麼）— 修改理由

這四個維度構成電子紀錄的「完整可重建性」——即使事後質疑，也能追溯整個事件鏈。

QU 職責：不僅啟用，還要選擇欄位

TR80 明確要求 Quality Unit 負責：

  • 確保稽核追蹤功能「全時開啟」
  • 評估各類 Metadata 的捕捉內容
  • 選擇驗證資料可靠性所需的必要欄位

此設計應配合批次放行程序及定期審查計畫共同規劃。

6.3.9.1 System-Level Audit Trail 6.3.9.1 系統層次稽核軌跡

原文內容 Original Text

If a system-level audit capability exists, the audit trail should capture, at a minimum: any attempt to log in, successful or unsuccessful; ID, date, and time of each login attempt; date and time of each logoff; device used; and function(s) performed while logged in, i.e., applications that the user attempted, successfully or unsuccessfully, to perform.

System audit trails cover the activities related to system policy changes, user activities (login, logoff, unauthorized logins and user privileges changes), changes to projects (creating, deleting, modifying and restoring), verification of project integrity and changes to systems. The Quality Unit should review system audit trails on a periodic basis established by the firm for any type of deletions.

系統級稽核追蹤：監控「誰」進出系統

System-Level Audit Trail 是最底層的安全防線，捕捉所有使用者認證事件：

  • 成功登入 / 登出
  • 失敗登入嘗試（含次數）
  • 身份確認失敗

  • 專案建立、刪除、修改、還原

類比：大樓門禁系統的刷卡記錄

  • Administrator 操作（尤其刪除）需雙人授權
  • 系統時鐘應與網路時間伺服器同步，防止時間篡改
6.3.9.2 Application-Level Audit Trail 6.3.9.2 應用程式層次稽核軌跡

原文內容 Original Text

Application-level audit trails monitor and log user activities, including which data files were opened and closed, and what specific actions have been taken, such as reading, editing, processing, and deleting records or fields, and publishing reports. This could also be called a project-level audit trail. Some applications may be sensitive enough to create an audit trail that captures "before" and "after" information for each modified record or the data changed within a record.

導師解析 Tutorial Commentary

應用程式層稽核：捕捉「操作行為」

Application-Level Audit Trail 比 System-Level 更細緻，記錄使用者在軟體應用中的每個動作：開啟/關閉哪個資料檔案、讀取、編輯、處理、刪除欄位、發布報告。

重要特色：Before & After 捕捉——不只記錄「有人改了這個欄位」，還記錄「原本是什麼值」和「改成什麼值」。這是資料完整性調查時還原事件的關鍵。

類比：Word 的「追蹤修訂」功能

Application Audit Trail 就像 Microsoft Word 的「追蹤修訂（Track Changes）」：每次修改都清楚標示刪除了什麼、新增了什麼、誰修改、什麼時間。主管可以接受或拒絕每一項修改，並保留完整歷程。

與 System-Level 的分工

System-Level 與 Application-Level 是互補的：

  • System：登入登出等系統事件，Application 不會捕捉的資訊
  • Application：檔案操作、資料修改、發布行為，System 不追蹤的內容

調查時兩者必須交叉比對，才能構成完整的時間軸。

查核觀察重點

查核官會尋找：是否有資料在「未公告的時間」被開啟或修改，特別是在結果不理想後不久發生的操作。Application Audit Trail 的時間序列往往能還原「是否有人試圖選擇性報告」。

6.3.9.3 Method Audit Trail 6.3.9.3 方法稽核軌跡

原文內容 Original Text

Method audit trails track the changes made to the method and typically contain the following information:

  • Method modification history
  • Method used for all injections
  • Method modified between the sequences

導師解析 Tutorial Commentary

方法稽核追蹤：確保方法一致性

Method Audit Trail 回答的核心問題是：「這個序列執行時，用的是哪個版本的方法？」

若方法在序列中途被修改（method modified between sequences），此追蹤將清楚顯示前後使用的方法版本，確保結果可歸因到正確的方法條件。

常見的 DI 風險點

方法稽核追蹤最常揭露的問題：

  • 分析師在同一序列中間靜默修改積分方法（e.g., 調整積分視窗使峰形看起來更好），若無版本追蹤則難以偵測
  • 使用未驗證或未正式核准的方法版本進行分析
  • Method ID 不一致（方法名稱相同但參數已更改）

審查建議

在批次放行審查中，應比較：Method Audit Trail 中的方法版本號 vs. 該批次 SOP 指定的核准方法版本。若版本不一致，必須調查並說明。

6.3.9.4 Results Audit Trail 6.3.9.4 結果稽核軌跡

原文內容 Original Text

Chromatograms or reports can serve as a results audit trail. Figure 6.3.9.4-1 is a model results audit trail report that presents integration type, date acquired, date processed and by whom, and if the chromatogram was altered after generating data. Typically, chromatograms should contain:

  • Sample description
  • Date and time of sample acquisition
  • Date and time of results processed, if possible
  • Time printed, published, or saved
  • Minimum method parameters, i.e., wavelength, injection volume, and method name/ID
  • Integration type
  • Retention time (any other system suitability parameters)
  • Integration events
  • Sample name change after execution

Filters can also be used to search the requirement from huge amounts of data. Figure 6.3.9.4-2 represents a filter to search whether the sample has been injected in multiple projects. For older systems with limited functionality, the reviewer should sign and stamp results printed on paper and review the electronic data according to the Quality Unit policy. If the results are published electronically, the reviewer should e-sign the results after reviewing and lock the signed file. If it will be used for any investigational purpose, the Quality Unit must review the electronic data per the firm's policy, as electronic data is deemed to be final. Once the printed data is audited, original electronic data should be available for reference and should not be altered. Any modification to the approved data needs to be justified.

導師解析 Tutorial Commentary

Results Audit Trail：「結果有沒有被動過手腳？」

Results Audit Trail 是最直接回答「結果是否被竄改」的工具。關鍵欄位包括：

  • Altered 欄：「Yes」表示樣品名稱在採集後被修改過——這是一個警示訊號
  • Date Acquired vs. Date Processed：時間差是否合理？
  • Integration Type：使用的是自動積分還是手動積分事件（manual integration events）？
  • Result Number：是否有跳號（gap）或重複——暗示刪除或複製

類比：醫院病歷的修改記錄

Results Audit Trail 就像電子病歷的修改紀錄：每次修改都標記「修改人、修改時間、修改前後的內容」，確保病歷的「Altered = No」代表從未被動過。

Filter 搜尋的重要性（Figure 6.3.9.4-2）

TR80 提到可以使用 Filter 搜尋「同一樣品是否在多個專案中被注射」。這是偵測試驗性注射（trial injections）最有效的方法之一：

  • 搜尋條件：同一樣品名稱（Sample Name）出現在多個 Project/Sample Set 中
  • 若發現，必須調查：是正當的重複測試，還是不當的「先試看看」？

電子資料為最終版本（Final Data）

TR80 明確：電子資料是最終紀錄，紙本只是副本。即使結果已用紙本簽核，若與電子資料有出入，以電子資料為準。

任何對已核准資料的修改，必須有正當理由，並留存記錄。

批次審查時的 Results Audit Trail 查核要點

  1. 確認 Altered = No（或若有 Yes，調查原因）
  2. 比對 Date Acquired 與 Date Processed 的時間邏輯
  3. 確認使用的 Integration Type 符合驗證方法
  4. 搜尋同樣本是否出現在其他專案（Filter 功能）

導師解析 Tutorial Commentary

序列稽核追蹤：「這個批次的注射序列有沒有被動過？」

Sequence Audit Trail（樣品集稽核追蹤）追蹤的是整個分析批次的執行順序完整性。它回答：

  • 序列執行過程中，是否有樣品名稱被更改？
  • 序列是否曾被中斷（aborted）？
  • 序列執行中是否有注射被插入或移除？

「中斷序列（Aborted Sequence）」的查核意義

一個序列在中途被中斷，最常見的原因包括：

  • 系統套適性（System Suitability）失敗，需要重新進針
  • 樣品前處理有問題
  • —— 或者 —— 選擇性地中止「結果不理想」的序列

查核官會要求解釋每一個被中斷的序列，並確認有對應的偏差報告（Deviation Report）。

審查重點

序列稽核追蹤與 Results Audit Trail 和 Method Audit Trail 應交叉比對，確保：

  • 序列執行期間的方法版本與 Method Audit Trail 一致
  • 所有注射結果都在 Results Audit Trail 中有對應記錄
  • 無「靜默刪除」的注射（Injection IDs 無跳號）

6.3.9.7 Injection Log 6.3.9.7 注射紀錄

原文內容 Original Text

Chromatographic data acquisition software typically has the ability to maintain a log of injections. In addition to the date and time, the software may include additional information associated with the injection such as lot number, blank or system suitability, reference, or sample. An audit trail ensures the integrity of the data and, if the audit trail is enabled, review of the injection log is not necessary. Injection log reports are typically built in by the software developer, though with limited ability to customize without expert programming capability. The type, ease of log retrieval, and ease of printing/publishing are also built-in functionalities of the software; however, their inclusion will vary among software suppliers. An injection log is separate and distinct from an audit trail; while it is a good tool to have, it is not essential to ensuring data integrity.

審查建議

6.3.10 Transactional Log / System Errors 6.3.10 交易紀錄 / 系統錯誤

原文內容 Original Text — 6.3.10.1 Chromatography

The transactional log (which may be referred to by other terms depending on the equipment vendor) and system error logs are online, instantaneous features that display pop-up messages about system functionality, user activity, and hardware-related issues or errors. Figure 6.3.10.1-1 shows the types of errors generated by the software or transactional log and a typical representation of a software transactional log. Figure 6.3.10.1-2 represents a typical transactional error log.

The transactional log is neither an audit trail nor is it intended to be a replacement for or component of other audit trails. A transactional log generally provides some additional information related to software (e.g., missing vial, lost prime) or hardware malfunctions (e.g., HSS fault, lost connection) and can help in interpreting the audit trail.

No one should have access or authority to manually change this log; however, the system can be configured to automatically purge the messages on a periodic basis to ensure efficient operation of the system processing memory. If the audit trail is never turned off, any deletion, modification, or copying of messages performed by the administrator will be recorded in validated audit trails (e.g., system audit trail). The Quality Unit should verify when a system or run has been interrupted due to a disconnection or power loss.

The messages appearing in the log may come from the application software, third-party software (e.g., Oracle database supporting system for chromatographic software) or other connected instruments (e.g., balance connected to HPLC) or equipment. Some chromatographic software packages offer this functionality. The transactional logs are system-level messages, temporarily stored and often automatically purged by the system at time-based defined intervals; their utility is therefore time-sensitive. These transactional logs may prove beneficial for trending (e.g., trending of most frequent instrument or processing errors that require attention helps in troubleshooting) or investigational purposes (e.g., describing the cause of the failure) and companies may utilize this information accordingly.

During software validation, messages will present as information, warning, or error according to listed categories (e.g., general, security). Critical messages and actions regarding data manipulation or data deletion that may appear in the transactional log must be captured in validated audit trails (e.g., system, result, sequence or sample, or method audit trails).

Categorization of error messages having an impact on the software and product ideally would be incorporated during software development and validation by the vendor. Some errors with titles that sound critical (e.g., cable disconnected, connection lost, communication failure) may not be captured in a validated audit trail but recorded in a transactional log. It may be difficult to confirm that these types of messages are all caused by intentional interruptions or have any impact on product quality data. It is therefore important to have appropriate controls and procedures in place to ensure that true power outages be recorded, especially if a chromatographic run is affected.

The Quality Unit for the lab must establish and validate error messages during equipment installation and qualification. Some error messages are specific to the operating system of the software and are not directly related to data or equipment operation. It is important to work with the software supplier to understand the description of messages that are recorded in the transactional log as they may be subject to evaluation during inspection. Further, it is important to identify those messages that are critical, i.e., related to data and instrument operations. For existing or previously installed equipment (e.g., legacy systems), during installation and qualification, the Quality Unit should assure that all transactional log messages are reviewed and understood, and that critical messages are identified and included in validated audit trails. Transactional log messages that have no impact on analyses or quality attributes of a product and messages that are also recorded in validated audit trails need not be retained.

For example, if a cable is disconnected from an HPLC to a LAC/E box, then the data will not be captured, and the transactional log will show a message as system interruption due to cable disconnection. Further dialogue between industry, health authorities, and vendors is needed to resolve how to address this evolving topic.

類比：汽車行車記錄器 vs. 法院公證文件

Transactional Log 像行車記錄器：持續記錄當下發生的事（Lost prime、Cable disconnected），但儲存空間有限，老資料會被覆蓋。
Audit Trail 像法院公證的文件：一旦簽署就永久有效，任何修改都要留下完整記錄，具有法律效力。

Figure 6.3.10.1-2 的 趨勢觀察重點

觀察「Column heater door open」在11月27日一天內出現五次：

  • 這可能代表儀器維護問題（門的彈簧損壞？）
  • 可能代表分析師頻繁開蓋進行非常規操作
  • Transactional Log 的趨勢分析價值：捕捉此類高頻事件，輔助根本原因調查

若此類事件影響分析結果，必須在 validated audit trail 中也有對應記錄。

重要：「聽起來嚴重」但可能不在 Audit Trail 的錯誤

TR80 特別指出：一些名稱聽起來很嚴重的錯誤（cable disconnected, connection lost, communication failure）可能只出現在 Transactional Log，不在 validated audit trail 中。

這對資料完整性調查是個挑戰——無法確定這些「失聯」是否為蓄意人為中斷。廠商需與軟體供應商共同釐清哪些訊息必須納入 validated audit trail。

IQ/OQ/PQ 時的 Transactional Log 工作

QU 必須在設備安裝與確效期間：

  1. 審查並理解所有 transactional log 訊息
  2. 識別哪些訊息為「關鍵訊息」（影響資料或儀器操作）
  3. 確認關鍵訊息是否也被 validated audit trail 捕捉
  4. 不影響分析或品質且已在 audit trail 記錄的訊息，不需要額外保留

非色層儀器的 Transactional Log

Transactional Log 不只是 HPLC/GC 特有，所有自動化分析設備都可能產生：

  • X-ray Diffraction（XRD）— 晶型分析
  • Karl Fischer（KF）— 水分測定
  • 各類自動化天平、pH 計、分光光度計等

XRD "Deleted the journal entry" — 誤導性訊息案例

這是 TR80 提供的最具警示性的範例：XRD log 顯示「Deleted the journal entry」——字面意思像是刪除了資料，但實際原因是作業中斷（job interruption），軟體系統自動產生此訊息作為內部清理機制。

若查核官（或分析師）不了解這個系統，可能會：

  • 錯誤地認定存在資料刪除/篡改行為
  • ——或相反—— 把真正的問題訊息忽視，因為「之前都是這樣顯示的」

解決方案：QU 必須與軟體供應商共同建立「訊息字典」，明確每條訊息的真實含義。

KF Log 的 "Sample data live modified" 警告

KF log 中出現兩次 Warning: Sample data live modified——這表示在分析過程中，樣品資料被即時修改。需確認：

  • 是正當的分析調整（e.g., 重新計算），還是不當的人工干預？
  • 此事件是否也在 validated audit trail（如 results audit trail）中有對應記錄？

跨設備 Transactional Log 的管理建議

  1. 建立各設備的 訊息分類清單（Critical / Non-critical）
  2. Critical 訊息必須確保也出現在 validated audit trail
  3. 為審查員提供設備特定培訓，避免因不熟悉系統而誤判訊息
  4. 在 IQ/OQ 確效文件中記錄各訊息的含義與對應處置
6.3.11 Common Deficiencies 6.3.11 常見缺失

原文內容 Original Text

The following problems, listed with the respective ALCS components, are often encountered and commonly found in audits or cited in FDA Warning Letters: (Ref. 55)

Computers

  • Shared passwords
  • No control over data generated
  • Data acquisition date-and-time-stamp changes to alter actual date and time of results
  • No control of automatic software updates
  • Computer systems/software validation errors
  • Time synchronization across all equipment and computers in the laboratory
  • Inappropriate database protection in computerized systems
  • Unauthorized changes made by analysts
  • Altering or setting back the computer's clock or date and time of the chromatographic injection

Server

  • Lack of oversight by Quality Unit
  • Validation errors
  • Improper network mapping that leads to data transmission losses
  • Cloud systems not verified for data transmission and data losses

Equipment

  • Single injections or sample trial injections of test samples made directly on instrument
  • Stand-alone equipment connected to computer without server lacks controls, routine audit trail reviews, and full data retention capabilities that prevent analysts or other personnel from deleting data
  • Features not appropriately selected or engaged, e.g., provision for trial injections without being captured in audit trails (implies lack of understanding or verification by lab personnel)
  • Improper configuration of computers or storage devices that can lead to duplication or falsification of data (implies lack of training for IT personnel)
  • Data losses due to power outage without investigation
  • Altering system suitability to make it appear as if the sample failure was caused by an equipment malfunction

Off-the-Shelf Software

  • Data processing scenarios not validated
  • Data calculations not validated
  • Part 11 compliance assumed or not verified
  • Software version changes incompatible with older files

Data Handling

  • Trial injections of test samples
  • Data integration problems and compromises
  • Not reviewing/publishing/enabling audit trails
  • Not reporting incidents

FDA Warning Letter 常見缺陷：查核官的視角

Section 6.3.11 是整個 TR80 最具實務價值的一節——它直接列出了 FDA 在 Warning Letter 中反覆引用的缺陷類型。以下以「查核官問題」的形式重新框架。

電腦管控缺陷：查核官會問的問題

  • 密碼共用：「請展示您的帳號是否唯一，且沒有與任何人共用密碼的記錄。」
  • 時間戳記篡改：「系統時鐘是否與 NTP 伺服器同步？若電腦時間被修改，是否有 audit trail 記錄？」
  • 軟體更新無管控：「自動更新是否在 GMP 環境中被停用，且更新程序是否受 change control 管控？」
  • 資料庫保護不當：「使用者是否可以直接存取資料庫而不留下 audit trail？」

伺服器缺陷：QU 監管的盲點

  • QU 缺乏監管：伺服器的變更（例如定期清理、備份設定修改）是否需要 QU 核准？
  • Cloud 系統未驗證：若使用雲端儲存，是否已驗證資料傳輸完整性？是否有資料遺失的偵測機制？
  • 網路映射錯誤：網路拓撲是否有文件記錄，且資料路徑是否已在確效中驗證？

設備管控缺陷：最常見的 DI 違規

  • 單次試驗注射：分析師直接在儀器上進行「試打」，結果不好就不存入系統——這是最典型的選擇性報告行為
  • Stand-alone 設備無稽核追蹤：未連接伺服器的設備往往可以直接刪除資料，形成管控盲點
  • Trial injection 功能未被 Audit Trail 捕捉：某些 ALCS 功能允許「非正式」注射，但若未納入稽核，等同於製造資料缺口
  • 修改系統適性以掩蓋樣品失敗：這是最嚴重的詐欺行為之一——把樣品失敗偽裝成設備問題

現成軟體（Off-the-Shelf）的合規陷阱

  • Part 11 合規「假設」：供應商宣稱軟體符合 Part 11，但廠商從未自行驗證——查核官會要求看驗證文件
  • 軟體版本升級：新版本是否能開啟舊版本建立的檔案？若不相容，舊資料的存取性（accessibility）受影響
  • 計算未驗證：ALCS 中的計算公式（e.g., 含量計算）是否有正式驗證測試案例？

資料處理缺陷：自我審查清單

對照 Section 6.3.11 的 Data Handling 缺陷，建議每季自我審查：

  1. 是否有任何「試驗性注射」記錄未納入正式資料集？
  2. 是否有積分問題被「私下處理」而非提報 OOS/偏差？
  3. 所有分析序列的稽核追蹤是否已在 Publishing 前被審查並啟用？
  4. 是否有任何事件（設備異常、資料遺失）未被正式記錄？

以上任何一項「是」，都代表潛在的 Warning Letter 風險。

本節重點回顧 Section Summary — 6.3.8 to 6.3.11

  • Publishing = 結果的正式鎖定步驟，發布後不可在無痕跡情況下修改；再處理需附理由說明 (§6.3.8)
  • 七大稽核追蹤層級（System → Application → Method → Results → Sequence → Project → Injection Log）各有分工，需交叉比對以確保完整覆蓋 (§6.3.9)
  • System Audit Trail：捕捉登入/登出/失敗登入/權限變更/專案刪除，定期審查，特別關注刪除事件 (§6.3.9.1)
  • Results Audit Trail：Altered 欄位是關鍵標記；電子資料為最終版本；Filter 可搜尋跨專案重複注射 (§6.3.9.4)
  • Project Audit Trail：涵蓋最廣，必須在專案建立時啟用；可偵測被刪除的注射記錄 (§6.3.9.6)
  • Transactional Log ≠ Audit Trail：前者是暫時性系統訊息（可被清除），後者是永久驗證紀錄；關鍵訊息必須同時出現在 validated audit trail (§6.3.10)
  • 誤導性訊息（如 XRD "Deleted the journal entry" 實為作業中斷）：QU 需與供應商建立訊息字典，培訓審查員 (§6.3.10.2)
  • FDA Warning Letter 高頻缺陷：密碼共用、時鐘篡改、試驗注射、稽核追蹤未啟用、系統適性造假——建立系統性自我查核清單可有效預防 (§6.3.11)
